# 功能模块 6：单章写作

> **功能编号**：6
> **功能定位**：按四段式结构创作章节，控制节奏和字数，自动集成记忆系统维护长篇连贯性。

---

## 一、核心特性

| 特性 | 说明 |
|------|------|
| **四段式结构** | 开篇、铺垫、高潮、收尾 |
| **节奏控制** | 根据情节类型调整节奏 |
| **字数控制** | 字数范围从立项定位单读取（详见步骤0） |
| **记忆系统集成** | 自动维护长篇连贯性 |

---

## 二、执行流程

### 阶段一：章节生成前（自适应上下文组装）

```
【执行步骤 — 自然语言记忆系统 v5.0】

[阶段一] 上下文组装
0. 检查记忆系统状态
   - 确认 05_24 写前预检已完成（检查进度看板或 `.done` 标记中是否有本章预检记录）
   - 若 05_24 预检未执行 → 先加载 `modules/05_24_功能模块.md` 完成写前预检，再回到本步骤
   - 若记忆系统目录不存在 → 05_24 预检会执行写前状态核对并报告缺失项；若 bible/ 下 5 个文档完全缺失（首次写前），05_26 写后流水线的前置检查会触发 05_18-A 降级初始化。本章写作本身不受阻断（初始化由写后流水线负责）
   - 预检通过后 → 继续以下步骤

1. 【章节类型判定】根据章节规划表和当前进度，判定本章类型：

   | 类型 | 特征 | 上下文策略 |
   |------|------|-----------|
   | 🔥 高潮章 | 爽点释放/大战/关键转折 | 加配：+5 额外摘要 + 所有相关伏笔原始摘要 |
   | 📜 伏笔回收章 | 计划回收 ≥1 个远古伏笔 | 专项：回读伏笔首现章摘要 + 上次推进章摘要 |
   | 🌊 过渡章 | 日常/铺垫/支线展开 | 减配：核心上下文即可（见步骤 3 过渡模式） |
   | 📦 标准章 | 常规推进 | 默认配置 |

   判定依据：
   - 章节规划表标注的「章节类型」和「爽点类型」
   - chekhovs_gun.md 中状态为 🔄（准备回收）的伏笔
   - 上一章结尾的悬念强度

1.5. 【资源预加载 — 根据章节类型自动匹配参考资源】
   基于步骤1的判定结果，按以下硬性规则加载参考资源（不可跳过）：

   | 判定条件 | 强制加载资源 |
   |----------|-------------|
   | 章节类型 = 开篇冲突 **且章节编号 ≤ 5** | `references/黄金开篇技巧.md` |
   | 爽点类型字段 ≠ 空 | `references/爽文创作理论.md` + `references/爽点设计策略.md` |
   | 章节类型 = 高潮爆发/大高潮 且 冲突类型 = 战斗冲突 | `references/打斗场景创作指南.md` |
   | 章节类型 = 对话/对谈为主 | `references/风格技巧详解.md` |
   | 章节类型 = 冲突反转 | `references/爽文创作理论.md` |
   | 世界观涉及AI/系统/游戏元素 | `modules/05_11_功能模块.md`（AI/系统流创作指南） |

   多条规则可同时触发（如：开篇首战章同时匹配开篇+爽点+打斗）。
   加载的资源内容将在步骤3上下文组装时纳入写作指导。

2. 【上下文预算】组装前估算总量，确保聚焦：
   - 目标：上下文总量控制在 100KB 以内（1M 上下文 agent 可放宽至 500KB；32K及以下模型降至 30KB）
   - 如果估算超限，按优先级裁剪：
     ① 砍远古 volume（保留最近 3 个）→ ② 减少额外摘要（5→3）→ ③ character_arcs.md 仅读活跃角色
   - 过渡章默认使用减配模式（见步骤 3），不触发预算检查

3. 【核心上下文组装】（RTCO 分级：P0 核心 → P1 重要 → P2 参考）

   P0 核心（所有章节必加载）：


   a) 故事圣经快照 [P0]
      → 读取 [P0] story_bible.md 的「当前状态快照」+「上一阶段梗概」

   P1 重要（标准章加载，过渡章减配）：

   b) 人物弧线（按权重 + 章节类型分级加载）[P1]

      【权重分级规则】
      - 主角级（96-100）：全量上下文（完整演化线 + 当前状态）
      - 核心配角（81-95）：当前状态 + 最近 10 章演化
      - 关键角色（61-80）：当前状态段落
      - 次要及以下（≤60）：仅姓名 + 状态摘要（1 句），除非本章为核心出场章节

      【章节类型调整】
      - 高潮章/伏笔回收章：核心配角升级为全量加载
      - 过渡章：核心配角降级为当前状态；关键及以下仅保留出场角色
      - 标准章：按权重默认规则

      【增长控制】
      - 如果 character_arcs.md 超过 30KB：
        * 将 500 章前的演化线压缩为每个角色一段「远古摘要」（3-5 句）
        * 原始细节下沉到对应 volume 压缩文件中，按需回读

   P2 参考：

   c) 剧情时间线（聚焦近期）[P2]
      → 读取 plot_timeline.md 的最近 20 章事件因果链
      → 检查是否有支线超过 50 章未推进（标记提醒用户）

   d) 伏笔清单 [P2]
      → 读取 chekhovs_gun.md 全部活跃伏笔
      → 筛选：状态为 🔄 准备回收 的 → 标记为本章优先处理
      → 筛选：回收窗口已过期（当前章 > 预期回收章 + 10）的 → 标记 ⚠️ 提醒用户

4. 【章节类型专属加载】

   🔥 高潮章加配：
      - 额外读取最近 10 章摘要（而非默认 5 章）
      - 读取所有涉及角色的完整弧线
      - 读取最近 2 个 volume 压缩（宏观历史回顾）

   📜 伏笔回收章专项加载：
      - 对每个计划回收的伏笔，读取其首现章节的原始摘要
      - 如果伏笔在多章中被推进，读取最近一次推进章节的摘要
      - 回顾首现章的场景氛围、人物情绪、对话细节
      - 确保第 500 章回收第 1 章伏笔时拥有完整语境

   🌊 过渡章减配：
      - 只读 story_bible 状态快照 + 最近 3 章摘要
      - character_arcs 仅读本章出场角色
      - 跳过 volume 加载
      - 跳过伏笔详细审查（仅确认无意外回收）

5. 【Volume 智能读取】
   - 默认读取最近 5 个 volume 压缩文件（排除 _template.md）
   - 如果伏笔涉及更早的 volume → 按需回读相关 volume
   - 如果 volume 总数超过 10 个 → 最近 5 个全读 + 其余仅在伏笔涉及时按需加载
   - 过渡章：仅读取最近 1 个 volume（宏观背景）

5.5 【YAML参数快照加载】（05_24 产出，辅助了解角色成长曲线和资源趋势）
   - 如果存在 `[书名]/记忆系统/phases/` 下的 `phase_*_params.md` → 读取最近 1 个 phase_params（角色成长曲线 + 资源消耗趋势）
   - 如果存在 `[书名]/记忆系统/volumes/volume_*_params.md` → 读取最近 1 个 volume_params（卷级参数快照）
   - 如果不存在（项目初期，尚未触发10章压缩）→ 跳过，不阻断
   - 此数据用于辅助上下文理解，不替代 bible 文档

6. 【风格同步】
```

### 阶段二：章节生成

```
[阶段二] 章节生成
0. 【读取字数配置】
   - 读取 [书名]/设定/[书名]_立项定位单.md，提取「每章字数」字段
   - 如果读到 → 字下限=要求值，字上限=要求值+500
     * 例：用户要求4000字 → 范围 4000-4500
   - 如果未读到（旧项目无此字段）→ 默认 2000-2500
   - 将字数范围记入变量，供后续步骤使用
   - 同时检查 [书名]/记忆系统/bible/story_bible.md 的「每章字数」字段作为备用
   - **冲突解决**：若立项定位单和 story_bible.md 都有值且不一致 → 以立项定位单为准（用户显式设定优先于记忆系统自动维护值），同时**自动纠正** story_bible.md 中的每章字数字段为立项定位单的值，确保两处数据一致

0.5. 【加载蓝图】
   - 读取章节规划表中本章的「出场角色」「场景」「冲突目标」字段
   - 确认蓝图要求的角色/地点/冲突在上下文中可用
   - 如果蓝图缺失 → 基于上一章摘要自动推断本章目标

```
【执行步骤】

1. 确认章节信息
   - 章节编号
   - 章节名称
   - 预期字数：[字下限]-[字上限]字（根据步骤0从立项定位单读取的配置。不足下限必须扩写，超过上限必须精简）

2. 加载四段式结构模板
   - 模板文件：templates/单章结构模板.md
   - 结构：开篇(10-15%) → 铺垫(20-30%) → 高潮(40-50%) → 收尾(15-20%)

3. 生成章节正文
   - 基于记忆包的上下文
   - 参考大纲或章节规划
   - 按四段式结构创作
   - 控制字数在预期范围内

4. 风格模仿（如用户指定）
   - 如果用户要求模仿特定作家风格 → 读取 `references/风格技巧详解.md`
     * 若该作家在速查表中 → 直接应用其句长/爽点/金句特征
     * 若不在表中 → 用三步分析法（句长指纹→爽点公式→金句模式）自行拆解
   - 将风格特征融入本章创作
   - 对话密集：参考相关对话写作技巧资料
   - 动作场景：参考《打斗场景创作指南》
   - 情感描写：参考相关情感描写技巧资料

5. 保存章节正文
   - 文件格式：纯文本，第一行为章节标题，第二行起为纯正文
   - 文件路径：[书名]/正文/第XXX章_章节名.txt
   - 严禁包含元数据和Markdown格式标记

6. 【字数强制检查 — 脚本校验，不使用LLM计数】
   - 执行脚本统计字数（替代LLM人工计数，更准确更快）：
     python -c "import re,sys; t=open(sys.argv[1], encoding='utf-8').read(); print(len(re.findall(r'[\u4e00-\u9fff]', t)))" "[书名]/正文/第XXX章_章节名.txt"
   - 字数 < [字下限] → ❌ 不合格，必须扩写。分析哪一段可以展开描写（对话/场景/心理/动作），重新生成阶段二步骤3-5
   - 字数 > [字上限] → ❌ 不合格，必须精简。删除冗余描写和重复信息，重新生成阶段二步骤3-5
   - [字下限] ≤ 字数 ≤ [字上限] → ✅ 合格，继续步骤7
   - 此步骤不可跳过。不合格的章节不得进入读者审稿流程。
   > 注：本步骤是字数检查的权威来源。05_26 步骤1 和 05_24 一阶检查#1 均引用此结果。
   > ⚠️ .done_wordcount 标记由 05_26 步骤1 统一写入，05_06 不自行标记。

7. 字数检查通过后 → **立即加载 `modules/05_26_功能模块.md` 并执行其全部步骤（步骤1-9），不得跳过。** 详见 `modules/05_26_功能模块.md`
```

### 阶段三：章节生成后

> ⛔ **本阶段通过步骤7 将写后操作完全委托给 `modules/05_26_功能模块.md`**，05_06 自身不内联执行任何写后操作（圣经更新×5+摘要+主代理逻辑自检+7子代理审核+周期压缩+版本快照+元数据追加）。
>
> 05_06 在阶段二步骤6字数检查通过后 → 直接将字数统计结果传递给 05_26 写后流水线闭环。**05_06 自身不执行记忆系统维护、不标记任何 .done 文件、不生成摘要。**
>
> 详细的圣经更新字段与写法以 `modules/05_26_功能模块.md` 步骤3 为唯一权威定义。

## 三、正文文件格式规范

```
【文件名格式】
第XXX章_章节名.txt

【文件内容格式】
第一行：章节标题
第二行起：纯正文内容
```

**示例**：

```
第一章：觉醒

（正文内容开始...）
```

**注意事项**：
- **严禁**在文件开头添加元数据（如"本章字数：XXX字"）
- **严禁**使用Markdown格式标记（如#标题、**加粗**等）
- **严禁**在文件结尾添加额外说明（**限定：初始写作阶段**。流水线闭环后，05_26 步骤9 会在末尾通过 `---` 分隔线追加元数据块，此为流水线标准操作，不属于"额外说明"范畴）
- 文件必须是纯文本格式，仅包含章节标题和正文内容（流水线完成后的元数据块除外）

## 四、批量写章节流程

当用户请求写接下来N章时，**逐章执行完整流水线**（写作+字数+审核），每章独立闭环：

```
for i in range(N):
    章节编号 = 当前章节 + i
    执行阶段一：章节生成前（上下文组装）
    执行阶段二：章节生成（写作+字数检查）
    执行阶段三：章节生成后（圣经更新+摘要）
    执行写后流水线（7个子代理审核 → PASS/FAIL → 修复）
    → 确认本章通过后，再进入下一章
```

⚠️ **禁止一次性写完所有章节再统一审核。** 每章必须独立走完写作→审核→修复的完整闭环，确保问题在当下解决，不累积到后续章节。

## 五、记忆系统集成说明（v5.0 自然语言记忆）

本模块集成了自然语言记忆系统的全部流程，在章节创作前后自动执行：

| 能力 | 实现方式 | 触发时机 |
|------|----------|----------|
| **L4 上下文包组装** | 直接读取 `[书名]/记忆系统/bible/` 5个文档 | 每章创作前 |
| **单章摘要回填** | 写入 `[书名]/记忆系统/summaries/ch_NNN_summary.md` | 每章创作后 |
| **圣经更新（5文档）** | 追加/修改 `bible/` 下自然语言文档 | 每章创作后 |
| **阶段压缩** | 10章摘要 → 压缩为 `[书名]/记忆系统/phases/phase_N_chX-Y.md` | 每10章 |
| **卷压缩** | 5个phase → 压缩为 `[书名]/记忆系统/volumes/volume_N_chX-Y.md` | 每50章 |
| **风格/人物/伏笔一致性** | 由 `modules/05_26_功能模块.md` 子代理 C（读者审稿）+ F（质量检查）覆盖，本模块不再单列 | 每章写后（05_26 闭环） |
| **人物一致性** | 由 `modules/05_26_功能模块.md` 子代理 A（五道门禁）+ D（连贯性）独立审核，不再由主代理自检 | 每章写后（05_26 闭环） |
| **伏笔一致性** | 由 `modules/05_26_功能模块.md` 子代理 D（连贯性）+ 步骤2 主代理逻辑自检覆盖 | 每章写后（05_26 闭环） |

**v5.0 实现方式**：所有记忆操作由 LLM 通过 `write_file` 直接完成，以自然语言 markdown 存储，LLM 打开即可理解，无需任何解析或外部工具。
