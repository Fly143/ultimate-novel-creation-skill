## 🔴 严重问题（Critical — 3 项）

### 🔴 CR-01：README 版本徽章严重滞后（v8.0.15 CR-01 延续，已达 4 轮未根治）

- **位置**：`README.md` 第3行 vs `SKILL.md` 第15行
- **描述**：README 徽章显示 `version-8.0.15`，SKILL.md frontmatter 声明 `version: 8.0.17`。版本号跨 **2 个小版本**不一致（8.0.15 → 8.0.16 → 8.0.17）。此问题在 v8.0.12、v8.0.14、v8.0.15、v8.0.17 连续四轮审计中被发现，但每次修复都不彻底——v8.0.16 commit 声称修复了「README版本」，但 v8.0.17 又跳过了 README 更新。**手动更新 README 徽章的流程存在系统性缺陷**。
- **影响**：外部引用（GitHub 徽章、第三方工具）显示严重过时的版本号（落后2个版本），与技能实际版本脱节。
- **方案**：
  1. 立即更新 README.md 第3行：`version-8.0.15` → `version-8.0.17`
  2. 根治：在 `.git/hooks/pre-commit` 添加脚本，从 `SKILL.md` frontmatter 自动提取版本号同步到 README.md 徽章。每次提交前自动执行，彻底消除人工遗忘。

### 🔴 CR-02：SKILL.md 触发表中 `已更新人设` 触发词无对应处理逻辑

- **位置**：`SKILL.md` 第80行 vs `modules/00_智能启动协议.md` 第81行
- **描述**：SKILL.md 快捷命令表中未列出「已更新人设」「改伏笔」等触发词映射到功能18工作流C。00_智能启动协议第81行有「更新人设 / 改伏笔 → 工作流C」，但 SKILL.md 作为触发词权威来源的第80行附近仅有「看记忆」「导出」等命令，缺少「改伏笔」「编辑圣经」等具体触发词映射。
- **影响**：用户说「改伏笔」时，agent 可能无法路由到功能18工作流C。
- **方案**：在 SKILL.md 快捷命令表中补全功能18工作流C/F的触发词映射。

### 🔴 CR-03：自审悖论设计原则第5条矛盾 — 既说「修复分离」（完美方案已实施），又说子代理会触发自主修改

- **位置**：`SKILL.md` 第274-282行（自审悖论设计原则）
- **描述**：第274行说「修复分离：有 FAIL 时，启动独立修复子代理读报告并修复，而非主代理自己修」→ 第282行说「完美方案需要审核和修复也分开（已实施：修复由独立子代理执行）」。这与 05_26 步骤2 主代理逻辑自检的修复规则矛盾——步骤2 说「不通过则主代理直接修正文（属剧情逻辑，不需启修复子代理），修完重跑字数+本步骤」。两条修复路径存在关键差异：
  - 步骤2 的 FAIL：主代理自己修（修复路径A）
  - 步骤4-6 的 FAIL：独立修复子代理修（修复路径B）
  
  SKILL.md 的自审悖论原则笼统地说「修复已完全分离」，但未区分步骤2 vs 步骤4-6 的不同修复规则。用户/agent 阅读自审悖论原则后可能形成错误期望——认为步骤2 的 FAIL 也会由独立子代理修复。
- **影响**：阅读 SKILL.md 自审悖论原则获取「修复全分离」的印象后，在 05_26 步骤2 遇到执行差异时可能困惑。对实际执行无直接影响（05_26 有更精确的规则），但文档层面存在误导。
- **方案**：在 SKILL.md 自审悖论原则中区分两种修复路径：「步骤2（主代理逻辑自检）由主代理直接修复（需写作上下文）；步骤4-6（子代理审核）的修复由独立修复子代理执行（避免自审偏见）。」

### 🟡 MD-02：05_12 节奏诊断报告命名仍有双格式

- **位置**：`modules/05_12_功能模块.md` 第41-42行
- **描述**：（v8.0.15 MD-03 延续）流水线模式报告 `第NNN章_节奏诊断报告.md`，独立调用模式报告 `[书名]_节奏诊断报告_第XX章.md`。两种格式差异显著——文件名前缀（无 vs 书名）、章节号位置（前缀 vs 后缀）。在 `设定/` 目录下共存的两种命名方式增加搜索难度。相比之下，05_17 连贯性报告的两种格式差异较小（`第NNN章_连贯性报告.md` vs `连贯性检查报告_第XX章.md`）。
- **影响**：agent 或用户在 `设定/` 目录下 grep 诊断报告时可能遗漏其中一种格式。
- **方案**：统一两种模式的报告命名格式，建议全部使用 `第NNN章_节奏诊断报告.md`（流水线格式），独立调用模式追加 `_独立` 后缀标记。

### 🟡 MD-03：05_16 修订优化模块未集成 delegate_task

- **位置**：`modules/05_16_功能模块.md`（仅43行）
- **描述**：05_16（修订优化）是**唯一未要求 delegate_task 独立子代理**的质量类模块。其他诊断/质量模块（05_12、05_13、05_14、05_15、05_17、05_22、05_25）均有显式的子代理调用规范。05_16 由主代理直接执行修订检查和同步，缺少上下文隔离。虽然修订优化需要更多写作上下文（不能完全隔离），但至少应该在「情节逻辑检查」和「人物一致性检查」部分提供可选的子代理模式。
- **影响**：修订时可能产生「自己审自己」的自审偏见——修订同一个 agent 刚写的章节，检查标准不独立。
- **方案**：在 05_16 步骤4 中增加可选项「情节逻辑/人物一致性检查可通过 delegate_task 启动独立子代理执行」。同时模块篇幅应适当扩充（当前43行过于精简，缺乏详细检查维度和输出规范）。

### 🟡 MD-04：SKILL.md 文件路径约定中的 `delegate_task` 实操要点引用了错误的行号

- **位置**：`SKILL.md` 第264行（delegate_task 实操要点）
- **描述**：第264行说「主代理在调用 delegate_task 前，必须将 context 模板中的占位符替换为实际项目路径值」，但未明确列出所有占位符类型（`{工作目录}`、`{书名}`、`{章节编号}`）和各自的取值规则。HI-05 在 v8.0.14 中被发现并在 v8.0.15 被标记为「✅ 已自然解决」，但实际上 SKILL.md 的 delegate_task 实操要点中**仍然没有**占位符替换的详细映射表。各子代理模块中 context 包含 `{工作目录}`、`{书名}`、`{章节编号}` 等占位符，但没有统一文档定义这些占位符的替换规则。
- **影响**：不同 agent 可能将 `{工作目录}` 替换为不同的路径（绝对路径 vs 相对路径），导致子代理找不到文件。
- **方案**：在 SKILL.md delegate_task 实操要点中增加占位符映射表：`{工作目录}` → 当前工作目录的绝对路径，`{书名}` → 从 story_bible.md 读取的书名，`{章节编号}` → 当前章节的3位零填充编号。
