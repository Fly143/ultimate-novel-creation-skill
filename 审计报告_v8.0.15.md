## 🔴 严重问题（Critical — 2 项）

### 🔴 CR-01：README 版本徽章未随 v8.0.15 更新

- **位置**：`README.md` 第3行 vs `SKILL.md` 第15行
- **描述**：README 徽章显示 `version-8.0.14`，SKILL.md frontmatter 声明 `version: 8.0.15`。Git commit `c5318dc` 消息称「第十轮全部修复(🔴5) — README版本」，但实际 README 文件**未更新**——徽章仍为 8.0.14。此问题已是第三次出现（v8.0.12→v8.0.14→v8.0.15），表明手动更新 README 徽章流程有系统性缺陷。
- **影响**：外部引用（GitHub 徽章、第三方工具）显示过时版本号，与技能实际版本脱节。
- **方案**：更新 README.md 第3行徽章 `version-8.0.14` → `version-8.0.15`。建议在 .git/hooks/pre-commit 添加脚本自动同步版本号，根治此问题。

### 🔴 CR-02：05_25 C 风格一致性缺乏整体 FAIL 覆盖规则

- **位置**：`modules/05_25_功能模块.md` 第70行、`modules/05_13_功能模块.md` 第108-112行
- **描述**：v8.0.15 在 05_13 F 中新增了「风格质量检查（独立评分维度，可触发 FAIL）」——这是正确的方向。但：
  - 05_25 C 中风格一致性（第70行 ★风格一致性）仅为「吸引力与可读性」维度下的一个子项，无独立评分权重，无「风格FAIL → 整体FAIL」的覆盖规则。
  - 05_26 步骤7 的 FAIL 检测涵盖 A/C/D/E/F/G 六项：F 的风格FAIL 可触发整体 FAIL（因为 F 已明确「任一子项不通过 → 整体 FAIL」），但 C 的风格子项 FAIL **不独立触发**——如果 C 的总分因其他维度高分而≥70（合格），风格问题不会触发修复。
  - C（读者审稿）和 F（质量）的风格视角不同：C 侧重「读者体验层的文风统一感」，F 侧重「技术层的语言风格/叙述视角/文风成熟度」。如果 C 发现风格问题而 F 未发现（比如：文风不突兀但「读起来感觉不对」），该问题会被漏掉。
- **影响**：读者视角的风格一致性问题可能被主代理忽略——C 的整体 PASS/FAIL 判定可能因其他维度高分而掩盖风格子项 FAIL。
- **方案**：在 05_25 C 子代理 context 中增加：「风格一致性子项评分为 0 → 整体 FAIL（无论其他维度分数）」。或在 context 中要求 C 输出「风格一致性独立评估：PASS/FAIL」，由 05_26 步骤7 额外检查此项。

### 🟠 HI-02：05_26 步骤2 .done_logic_selfcheck 标记延迟写入（v8.0.14 HI-04 延续）

- **位置**：`modules/05_26_功能模块.md` 第64-82行（步骤2）、第159行（步骤7标记写入）
- **描述**：步骤2 说「自检通过则继续」，步骤7 才写 `.done_logic_selfcheck` 标记。步骤2 的「通过」和步骤7 的标记写入之间隔着步骤3-6 的所有子代理。如果 agent 在执行步骤3-6 过程中中断（网络/会话超时/用户打断），`.done_logic_selfcheck` 不会被写入，即使步骤2 已经通过。
- **影响**：跨会话恢复时，「补缺」流程无法判断步骤2 是否已完成，可能重复执行逻辑自检。虽然步骤2 纯主代理执行、不涉及子代理开销，但重复执行仍是浪费。
- **方案**：步骤2 通过后**立即**写入 `.done_logic_selfcheck`（不等步骤7），或改为步骤2 独立标记而非在步骤7 批量写入。

### 🟠 HI-03：05_24 写前预检与 05_06 阶段一记忆系统检查存在重复路径（v8.0.14 HI-03 延续）

- **位置**：`modules/05_24_功能模块.md` 第54-76行 vs `modules/05_06_功能模块.md` 第26-34行
- **描述**：00_强制执行协议步骤C 说加载 05_24 做预检（5字段核对），但 05_06 阶段一步骤0 又独立做降级初始化检查（「记忆系统不存在 → 自动调用 05_18-A」）。两条路径可能：
  - 先后都执行 → 重复工作（05_24 检查状态，05_06 又检查一次）
  - agent 只执行其中之一 → 另一条路径的检查被跳过
- **影响**：agent 可能跳过预检直接进入写作（如果它先加载 05_06 的阶段一），导致写前状态不完整。
- **方案**：统一入口——05_06 阶段一步骤0 改为「确认 05_24 写前预检已完成（检查进度看板或 .done 标记），若未完成则加载 05_24」。

## 🟡 中等问题（Medium — 5 项）

### 🟡 MD-01：05_01 步骤8.4 报告将强约束文件与 bible 混在同一节

- **位置**：`modules/05_01_功能模块.md` 第64-85行
- **描述**：步骤8.4 报告将【强约束系统】的三个约束文件列在【故事圣经】5个文件之后、同一报告块中，结构上暗示强约束属于故事圣经。实际约束文件位于 `记忆系统/constraints/` 而非 `记忆系统/bible/`，语义上两者是不同系统（事实记录 vs 风险分析）。
- **影响**：用户可能误以为强约束文件是故事圣经的一部分，造成概念混淆。对 agent 执行无实际影响。
- **方案**：将强约束系统文件移到报告中独立的「【强约束系统】」节，与「【故事圣经】」并列层次。

### 🟡 MD-02：05_06 阶段三注释「不执行任何写后操作」与步骤7「立即加载05_26」存在语义张力

- **位置**：`modules/05_06_功能模块.md` 第197行（步骤7）、第200-206行（阶段三注释）
- **描述**：阶段三注释说「本阶段不执行任何写后操作」，但步骤7 说「立即加载 modules/05_26_功能模块.md 并执行其全部步骤」。两者实际语义一致（步骤7 把写后委托给 05_26），但如果 agent 先读到阶段三的声明再回溯步骤7，可能产生「该不该做写后」的判断矛盾。
- **影响**：某些 agent 可能选择执行阶段三注释的字面意思（不执行任何写后操作），从而跳过步骤7 和整个写后流水线。
- **方案**：阶段三注释改为：「本阶段通过步骤7 将写后操作完全委托给 `modules/05_26_功能模块.md`，05_06 自身不内联执行任何写后操作。」

### 🟡 MD-03：05_12 节奏诊断报告命名有两种格式

- **位置**：`modules/05_12_功能模块.md` 第41-42行
- **描述**：流水线模式报告 `第NNN章_节奏诊断报告.md`，独立调用模式报告 `[书名]_节奏诊断报告_第XX章.md`。两个文件名格式差异显著（一个有 `[书名]_` 前缀、章节号位置不同），但都在 `设定/` 目录下。其他子代理（05_17 连贯性）也使用类似双格式，但至少格式上有规律。
- **影响**：用户/agent 搜索诊断报告时可能遗漏其中一种格式。
- **方案**：统一两种模式的报告命名格式，或在模块中显式标注两种格式的存在。

### 🟡 MD-04：06_场景资源映射的部分触发机制描述已过时

- **位置**：`modules/06_场景资源映射.md` 第10-15行
- **描述**：06 提到「生成前检查每章触发」和「生成后检查每章触发」，但 05_26 写后流水线已接管所有检查。06 中描述的触发时机与 05_26 步骤4-6 的实际触发方式不一致——06 说的是 agent 直接执行检查，实际是 delegate_task 子代理执行。虽然 06 已标注为「参考文档」，但其内容仍有误导性。
- **影响**：agent 如果按 06 的映射直接加载模板自己做检查，就绕过了子代理隔离机制。
- **方案**：在 06 的触发机制列增加「由 05_26 写后流水线通过 delegate_task 子代理触发」标注，或明确标明「运行时以 05_26 步骤4-6 为准」。

### 🟡 MD-05：05_25 审稿报告存储路径与其他子代理报告不一致

- **位置**：`modules/05_25_功能模块.md` 第46行 vs `modules/02_文档存储规范.md` 第80行
- **描述**：05_25 C 审稿报告写入 `记忆系统/summaries/ch_NNN_review.md`，而其他子代理报告（五道门禁/强约束/连贯性/节奏/质量/逻辑因果审查）均写入 `设定/第NNN章_xxx报告.md`。02_文档存储规范第80行对此有显式说明（⚠️ 标注），但 05_26 步骤5 子代理 C 的 context 中没有显式写出完整路径，子代理可能误写。
- **影响**：子代理 C 可能误将报告写入设定/ 而非 summaries/，导致「补缺」流程找不到审稿报告。
- **方案**：在 05_26 步骤5 子代理 C 的 context 模板中显式写出完整路径 `{书名}/记忆系统/summaries/ch_NNN_review.md`。
