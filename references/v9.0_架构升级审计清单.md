# v9.0 架构升级审计清单

> 适用场景：全能小说作家技能做重大架构升级后，全量扫描所有模块文件的旧架构残留。
> 核心原则：**变更集 ≠ 影响面**。修改了 3 个文件不代表只有 3 个文件需要审计。
> 教训：v8→v9.0 升级中，前四轮只审 3 个核心文件，第五轮全量扫描发现 6 个边缘文件共 14 处残留。

## 高危模式 grep 清单

```bash
# 子代理返回格式残留（应返回完整报告而非 PASS/FAIL 一行）
grep -rn "只返回一行" modules/
grep -rn "仅返回.*PASS" modules/

# 主代理逻辑自检残留（应已全部移除）
grep -rn "主代理逻辑自检" modules/ SKILL.md README.md

# 旧步骤编号（05_26 从 9 步缩为 8 步）
grep -rn "步骤1-9" modules/ SKILL.md
grep -rn "步骤9" modules/ SKILL.md

# 圣经更新步骤号（步骤3 → 步骤2）
grep -rn "步骤3.*圣经" modules/

# 旧架构术语
grep -rn "阶段三" modules/

# .done 标记（logic_selfcheck → logic_causal）
grep -rn "logic_selfcheck" modules/ SKILL.md README.md

# 子代理 B 步骤编号（步骤4 → 步骤3）
grep -rn "步骤4.*子代理B" modules/
```

## 审计策略

1. **先 grep，再按需读取**：30+ 个模块文件，不要逐个全读。先搜高危模式，只读命中文件
2. **变更说明 vs 真实残留**：区分「已移除/已合并」等变更说明（允许）和真实残留（需修复）
3. **连续两轮无🔴🟠 才停**：P70 铁律同样适用于架构升级审计
