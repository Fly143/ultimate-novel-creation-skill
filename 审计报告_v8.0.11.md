### 🟠-4：`delegate_task` 调用格式在模块间不一致
- **位置**：`modules/05_12_功能模块.md`、`modules/05_13_功能模块.md`、`modules/05_22_功能模块.md`、`modules/05_26_功能模块.md` 等多处
- **描述**：多个模块包含 `delegate_task` 调用模板，但格式不统一：
  - 05_12：使用函数调用语法 `delegate_task(goal:, context:, toolsets:)` 
  - 05_13：使用 YAML 语法 `delegate_task:\n  goal:\n  context: |`
  - 05_22：混合使用函数调用语法
  - 05_26：使用子弹列表 `goal: ...\ncontext: ...`
  - 05_25：使用函数调用语法
  - 05_14：使用函数调用语法
  - 05_15：使用函数调用语法
  
  `delegate_task` 是 Hermes Agent 的工具函数，实际调用时需用 `delegate_task(goal=..., context=..., toolsets=...)` 格式。YAML 语法可能在执行时被误解析，无法传递到工具调用。
- **修复**：
  - 统一所有模块中的 `delegate_task` 调用模板为 Python 函数调用语法
  - 或明确标注"以下为伪代码，实际调用需转换为工具调用格式"
  - 推荐：在 SKILL.md 或 00_强制执行协议中统一定义子代理调用规范，各模块引用该规范

### 🟠-5：05_12 子代理 context 中 `\{` 转义问题
- **位置**：`modules/05_12_功能模块.md` 第 24 行
- **描述**：`delegate_task` 的 `context` 字符串中使用了 `{书名}/正文/...` 等变量占位符。在 Markdown 代码块中这些是纯文本，但如果主代理在构造子代理调用时直接把模板字符串传入（不作变量替换），子代理会收到未替换的 `{书名}` 字面量，无法找到文件。需要在调用规格中说明：主代理在发送前须将 `{书名}`、`{工作目录}` 等替换为实际值。
- **修复**：在 SKILL.md 的自审悖论设计原则章节或 05_26 子代理调用规范中增加一条：
  ```
  主代理在调用 delegate_task 前，必须将 context 模板中的占位符
  （{书名}、{工作目录}、{章节编号} 等）替换为实际项目路径。
  ```
  同时在各模块的调用模板中加注释提醒。

### 🟡-2：05_06 硬编码 100KB 上下文预算
- **位置**：`modules/05_06_功能模块.md` 第 59 行
- **描述**：`上下文总量控制在 100KB 以内` 是硬编码值。不同模型的上下文窗口差异巨大（4K/8K/32K/128K/1M），100KB 对 4K 模型太大，对 1M 模型又过于保守。虽然注释提到"1M 上下文 agent 可适当放宽"，但没有给出放宽的具体策略。
- **修复**：将硬编码改为模型感知策略：
  ```
  上下文预算 = min(模型上下文窗口 * 0.3, 100KB)
  ```
  或根据当前使用的模型自动调整。至少提供三个档位：小模型(≤32K)/中模型(128K)/大模型(≥1M)。

### 🟡-3：规则集覆盖的流派不足
