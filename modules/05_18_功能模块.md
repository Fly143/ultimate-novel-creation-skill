### 18. 记忆系统管理（长篇连载专用）v5.0

> **触发条件**：用户开始长篇小说创作；用户说"建立记忆系统""检查记忆""修复跑偏"；跨会话恢复后需要确认记忆；手动查看或编辑故事圣经。
>
> **v5.0**：记忆系统采用**自然语言文档**。所有记忆以 LLM 可直接读写的 markdown 文件存储。

---

## 一、功能概述

v5.0 记忆系统由 4 层自然语言文档构成：

```
记忆系统/
├── bible/
│   ├── story_bible.md        # L1: 故事圣经（世界观/阶段划分/状态快照）
│   ├── character_arcs.md     # L1: 人物演化弧线（非静态人设卡）
│   ├── plot_timeline.md      # L1: 剧情时间线（事件因果链）
│   └── chekhovs_gun.md       # L1: 伏笔清单（契诃夫之枪追踪）
│   └── character_database.md  # L1: 角色数据库（出场角色基础信息/状态）
├── summaries/
│   ├── _template.md          # 单章摘要模板
│   ├── ch_001_summary.md     # L2: 第1章摘要
│   ├── ch_002_summary.md     # L2: 第2章摘要
│   └── ...                   # 每章一个
└── phases/
│   ├── _template.md          # 阶段压缩模板
│   ├── phase_1_ch1-10.md     # L3: 第1-10章压缩
│   ├── phase_2_ch11-20.md    # L3: 第11-20章压缩
│   └── ...                   # 每10章一个
├── volumes/
│   ├── _template.md          # 卷压缩模板（L3.5）
│   ├── volume_1_ch1-50.md    # 第1-50章卷压缩
│   └── ...                   # 每50章一个
```

## 二、核心工作流

### 工作流 A：项目初始化（首次新建小说）

**触发时机**：用户新建长篇小说，完成基本设定（书名、大纲、人设）后

**执行步骤**：

1. **创建目录并从 skill 复制模板**
   ⚠️ skill 内的 `memory-system/` 是模板，只读，永远不要直接修改。
   所有文件操作都针对 `[书名]/记忆系统/`（项目目录）。

   创建目录结构：
   ```
   [书名]/
   ├── 记忆系统/
   │   ├── bible/
   │   │   ├── story_bible.md       ← 从 skill 模板复制
   │   │   ├── character_arcs.md    ← 从 skill 模板复制
   │   │   ├── plot_timeline.md     ← 从 skill 模板复制
   │   │   ├── chekhovs_gun.md      ← 从 skill 模板复制
   │   │   └── character_database.md ← 从 skill 模板复制（templates/bible/character_database_template.md）
   │   ├── summaries/
   │   │   └── _template.md         ← 从 skill 模板复制
   │   ├── phases/
   │   │   └── _template.md         ← 从 skill 模板复制
   │   └── volumes/
   │       └── _template.md         ← 从 skill 模板复制
   └── constraints/
       ├── cheat_constraints.md     ← 从 skill 模板复制（templates/constraints/cheat_constraints_template.md）
       ├── time_constraints.md      ← 从 skill 模板复制（templates/constraints/time_constraints_template.md）
       └── narrative_threads.md     ← 从 skill 模板复制（templates/constraints/narrative_threads_template.md）
   ```

   操作：从 skill 的 `memory-system/` 读取各模板文件，写入 `[书名]/记忆系统/` 对应路径。
   constraints 模板从 skill 的 `templates/constraints/` 目录读取，写入时去掉文件名中的 `_template` 后缀。

2. **填充圣经文档**（修改的是 `[书名]/记忆系统/` 下的副本，非 skill 模板）
   - **story_bible.md**：填入书名、题材、世界观简版、预期字数
   - **character_arcs.md**：基于已有角色设定，填入初始状态
   - **plot_timeline.md**：填入大纲中的主线事件及支线
   - **chekhovs_gun.md**：填入初始伏笔（如有）
  - **character_database.md**：填入已出场角色的基础信息（角色名、首次出场章节、当前状态）

3. **输出初始化确认**
```
✅ 记忆系统已初始化：[书名]/记忆系统/
   ✓ story_bible.md（项目概况 + 世界观 + 阶段模板）
   ✓ character_arcs.md（人物演化弧线模板）
   ✓ plot_timeline.md（剧情时间线模板）
   ✓ chekhovs_gun.md（伏笔清单模板）
   ✓ character_database.md（角色数据库模板）
   ✓ constraints/（约束模板 ×3）

📋 状态：已就绪，可开始创作
```

### 工作流 B：查看记忆系统（查看当前状态）

**触发时机**：用户说"看看现在的记忆""检查人物状态""显示伏笔"等

**执行步骤**：

1. **按需读取圣经文档**
   - "看整体" → 读 story_bible.md（故事概括 + 当前状态快照）
   - "看某个人物" → 读 character_arcs.md 中对应角色的演化线
   - "看剧情" → 读 plot_timeline.md
   - "看伏笔" → 读 chekhovs_gun.md
   - "看角色库" → 读 character_database.md（已出场角色的当前状态摘要）

2. **输出示例**
```
📖 当前创作状态

【进度】第31章 / 目标300章（已完成10%）

【活跃人物】
- 主角·陈凡：筑基中期，位于北域冰原，目标是寻找寒冰异火
- 林雪：金丹初期，与陈凡同行，关系在30章确认道侣
- 神秘老者：第15章出场，身份未明，当前潜伏

【正在推进的剧情线】
- 主线「寻火之旅」：第25-30章已获得冰焰线索，正前往北域
- 支线「宗门的秘密」：第20章后暂未推进

【待回收伏笔（高重要度）】
- 陈凡的身世玉佩（第5章埋设，已暗示与北域有关）
- 林雪的诅咒（第15章埋设，至今未解释）

【未推进超过 20 章的伏笔】
- 神秘黑衣人的真实身份（第8章埋设）→ 考虑近期回收
```

### 工作流 C：手动编辑圣经

**触发时机**：用户说"帮我更新人设""改一下故事走向""标记几个伏笔已回收"

**执行步骤**：

1. **找到对应圣经文档**（bible/ 下的 4 个文件之一）
2. **按需修改**：
   - 人物变化 → 追加到 character_arcs.md 的对应角色线
   - 伏笔回收 → 更新 chekhovs_gun.md 状态为 ✅ 已回收
   - 世界观变动 → 更新 story_bible.md 世界观简版
   - 时间线调整 → 更新 plot_timeline.md
3. **保持自然语言格式**，不要写成 JSON 或结构化数据
4. **确认修改完成**：列出修改摘要

---

### 工作流 D：阶段压缩（手动触发）

**触发时机**：用户完成 10/20/30...章后；或用户说"压缩一下记忆"

**执行步骤**：

1. **读取最近 10 个单章摘要**（summaries/ch_NNN_summary.md）
2. **提取核心信息**：
   - 本阶段剧情主线
   - 每位角色的弧线变化
   - 已回收的伏笔
   - 新埋设的伏笔
   - 结尾状态
3. **写入压缩文档**：
   - 文件：[书名]/记忆系统/phases/phase_{序号}_ch{起始}-{结束}.md
   - 格式参考 phases/_template.md
4. **更新 story_bible.md 的阶段划分**
5. **输出压缩结果**
```
✅ 阶段压缩完成

【阶段1：第1-10章】→ phase_1_ch1-10.md
阶段标题：「初入修仙」
核心冲突：主角从废柴到觉醒
关键转折：第5章觉醒灵根，第8章第一次生死战

【圣经已同步】
✓ story_bible.md 阶段划分已追加
```

---

### 工作流 D2：卷压缩（每 50 章，手动触发或自动）

**触发时机**：用户完成 50/100/150...章后；或用户说"压缩五十章""做卷压缩"

**执行步骤**：

1. **读取最近 5 个 stage 压缩文件**（[书名]/记忆系统/phases/ 下的 phase_N_chX-Y.md ~ phase_N+4_chX-Y.md）
2. **提取全书级核心信息**：
   - 本卷主线推进概况（从…到…）
   - 主角弧线关键转变
   - 仍开放的高重要度伏笔
   - 已闭环的重要线索
3. **刷新伏笔近况**：
   - 打开 [书名]/记忆系统/bible/chekhovs_gun.md
   - 遍历所有活跃伏笔（高/中/低重要度），检查每条伏笔的「近况」字段
   - 根据本卷 50 章情节，更新每条伏笔的当前状态（1-2 句自然语言）
   - 特别注意：近况为空或超过 50 章未更新的伏笔 → 回读其首现章节摘要，补充近况
   - 近况字段不经过压缩链，确保第 800 章时仍能看到第 5 章伏笔的细节
4. **写入卷压缩文档**：
   - 文件：[书名]/记忆系统/volumes/volume_{序号}_ch{起始}-{结束}.md
   - 格式参考：[书名]/记忆系统/volumes/_template.md（项目初始化时从 skill 模板复制）
   - 控制在 500 字以内
5. **输出卷压缩结果**
```
✅ 卷压缩完成

【卷1：第1-50章】→ volume_1_ch1-50.md
概括：「废柴主角觉醒至宗门小比夺冠为止」
关键转变：尚未 → 筑基
开放伏笔：祖传玉佩（#1）、神秘老者（#3）
```

### 工作流 E：跨会话恢复

**触发时机**：新会话中用户说"继续写《XXX》""接着上次的来"

**执行步骤**：

1. **定位项目目录**
   - 如果用户提供了小说名称，在当前工作目录下定位 `[书名]/` 目录
   - 如果用户未提供名称，搜索 `*/记忆系统/bible/story_bible.md` 定位已有项目
   - 确认 `[书名]/记忆系统/` 目录存在
   - 如果找不到项目：提示用户提供小说名称或确认项目路径
2. **如果存在，加载恢复上下文**（智能加载，按需而非全量）：
   - 读取 story_bible.md → 了解项目概况 + 当前状态快照 + 最近 3 个阶段划分
   - 读取 character_arcs.md → 关注「当前状态」段落；如文件 >30KB 则远古部分已自动压缩为摘要
   - 读取 plot_timeline.md → 最近 20 章事件因果链
   - 读取 chekhovs_gun.md → 所有活跃伏笔，特别注意状态为 🔄 和 ⚠️ 过期的
   - 读取 character_database.md → 已出场角色的基础信息（角色名/首次出场/当前状态）
   - 读取最近 5 个 volume 压缩（非全量，排除 _template.md）
      → 快速补课全书宏观历史
   - 读取最近 2 个 phase 压缩（排除 _template.md）
      → 了解最近 20 章上下文（恢复场景需更广的连续性，比单章写作加载更多 phase）
   - 读取最近 5 个 summary → 了解最近细节
   - 本章计划回收伏笔 → 额外回读该伏笔首现章节摘要
3. **确认恢复完成**
```
✅ 项目恢复成功

【项目】《书名》— 已写至第 XX 章
【上次写到】
- 章节：第 XX 章
- 结尾悬念：...

【当前需要推进】
- ...

📋 准备好继续创作，请告诉我需求
```

### 工作流 F：紧急修复跑偏

**触发时机**：用户说"人物跑偏了""剧情对不上了""帮我检查一致性"

**执行步骤**：

1. **对比问题章节 vs 圣经记录**
   - 读取 character_arcs.md 中该角色的设定
   - 对比问题章节中该角色的言行
   - LLM 自行判断是否存在 OOC

2. **对比剧情时间线**
   - 读取 plot_timeline.md
   - 问题章节中的事件是否与之前的事件因果矛盾？
   - 物品/线索的状态是否一致？

3. **修复方案**
   - 轻微偏差：标记在圣经中，写下一章时纠正
   - 严重 OOC：建议改写问题章节
   - 剧情矛盾：更新圣经记录或修改时间线

4. **输出修复报告**
```
=== 一致性检查报告 ===

【人物检查】
✓ 主角陈凡：一致（当前心态符合第30章结尾的转变）
⚠️ 林雪：轻微偏差（本章态度比预期冷漠，注意解释）
✗ 张长老：第15章已退场，本章被错误提及 → 回退修改

【剧情检查】
✓ 主线时间线：一致
⚠️ 玉佩伏笔：原计划第35章回收，当前第32章已提前暗示 → 已更新 chekhovs_gun.md

【建议】
1. 第32章林雪冷漠态度：在33章通过回忆或对话解释原因
2. 张长老误提及：回退第32章，改为另一长老
3. 玉佩伏笔提前：已更新回收窗口为第33-34章
```

## 三、关键原则

1. **人类可读 > 机器可读**：所有记忆用自然语言，不用 JSON/字典/表格
2. **只存因果，不存统计**：不记录句长、词频等统计数据
3. **人物是弧线，不是静止标签**：记录主角从 A → B → C 的演变，而非"性格：勇敢"
4. **伏笔有开关**：每条伏笔必须标记已埋/准备/已收，没有"待定"
5. **阶段压缩和卷压缩不可逆**：原始摘要保留在 summaries/ 供回溯，压缩不删除原始数据
6. **跨会话可秒恢复**：新会话读取 bible + 所有 volumes + 2个阶段 + 5个摘要 即可恢复到停写时的知识状态
7. **LLM 自身是检查器**：LLM 自己负责文风判断/人物一致性/剧情逻辑检查
8. **模板与实例分离**：skill 内的 `memory-system/` 文件是模板（只读），项目初始化时复制到 `[书名]/记忆系统/` 后，所有后续读写都针对项目副本。永远不要修改 skill 内的模板文件，否则下次新建项目时模板已被污染。
9. **Bible 文档增长控制**：
   - character_arcs.md 超过 30KB 时自动压缩：将 500 章前的演化线压缩为每个角色 3-5 句「远古摘要」，原始细节下沉到对应 volume 压缩文件
   - story_bible.md 的阶段划分超过 20 个阶段时，将前 10 个阶段合并为一句概括
   - plot_timeline.md 的事件表超过 100 行时，将 500 章前的事件行合并为「远古事件摘要」
   - 压缩时输出变更报告，告知用户哪些内容被压缩、如何回溯
