## 📂 文件路径约定

**本 skill 中所有文件引用遵循以下规则：**

1. **skill 内部文件**：`modules/`、`references/`、`templates/`、`memory-system/`（skill 内模板）等路径，均相对于本 skill 的根目录。
   - 示例：`references/书名生成策略.md` → skill 目录下的 references/书名生成策略.md
   - 示例：`memory-system/bible/story_bible.md`（不带 `[书名]/` 前缀）→ skill 内模板文件，只读

2. **项目文件**：带 `[书名]/` 前缀的路径，相对于当前工作目录。`[书名]` 是项目初始化时用户确认的书名。
   - 示例：`[书名]/记忆系统/bible/story_bible.md` → 项目目录下的记忆系统副本
   - 示例：`[书名]/设定/[书名]_人设卡.md` → 项目目录下的设定文件

3. **记忆系统文件区分**：
   - `memory-system/xxx`（无前缀）= skill 内的只读模板 → 仅用于项目初始化时复制
   - `[书名]/记忆系统/xxx`（有前缀）= 项目目录下的读写副本 → 所有后续操作的目标

4. **项目查找**：跨会话恢复时，在当前工作目录下搜索 `*/记忆系统/bible/story_bible.md` 定位已有项目。

## ⚠️ 自审悖论设计原则（★v9.0 纯调度架构）

> **核心教训**：同一个模型、同一个对话里"自己写自己审"是无效的。
> 审核结果天然偏向"还行"——因为审核器和写手共享上下文，已经被"洗脑"了。

**v9.0 纯调度架构解决方案**：

1. **三重分离**：写作（05_06 子代理）、审核（05_26 子代理×7）、修复（05_26 修复子代理）全部由独立子代理完成
2. **主代理降为编辑**：仅保留记忆维护（圣经更新+摘要）+ 调度职责。不再亲自写作、不再自检逻辑、不再读审核报告
3. **审核→修复的零补课闭环**：审核子代理返回完整报告（含段落引用+矛盾点+修改建议），修复子代理**只读审核报告+正文**，无需重读记忆文档。避免修复循环中反复加载 bible/arcs/timeline 的 token 爆炸
4. **跨会话无差异**：无论是单次会话内连续写作还是跨会话续写，写作子代理的起点一致（都是从磁盘读文件），消除了 v8.x 中"主代理单次会话内的肌肉记忆优势"这一伪优势

**为什么跨会话场景下主代理没有优势**：
- 新会话启动 → 工作流E（跨会话恢复）→ 读 bible + volumes + phases + summaries → 重建上下文
- 这个过程和一个全新子代理做的事情**完全一样**：从磁盘读文件 → 组装理解 → 生成文本
- 唯一区别是这个"新脑子"恰好叫"主代理"而不是"子代理"
- 既然 200 万字不可能一镜到底，跨会话是常态，那全卸给子代理就没有额外成本

**不能脚本化的 vs 能脚本化的**：
- ❌ 伏笔判断、逻辑矛盾、描写充分度、节奏评估 → 必须靠模型（所以用独立子代理）
- ✅ 人名/地名/专有名词与设定列表的字符串匹配 → 可以脚本化（但目前未实现，优先级低）

**delegate_task 的边界**：
- ✅ 子代理的审核判断是独立的（没有写作偏见）
- ✅ 修复子代理不依赖记忆文档（审核报告自包含）
- ✅ 跨会话恢复无架构劣势（主代理和新子代理起点相同）
- ⚠️ 修复子代理完成后的二次审核仍需原审核子代理重跑（但只重跑 FAIL 的，不是全部 7 个）

**Pitfall**：子代理 `max_concurrent_children` 限制（默认3），不能一次起超过3个并行子代理。超过时需分批（先3个，完成后再3个）。

**delegate_task 实操要点**：
- 子代理的 `context` 必须包含：项目工作目录路径、章节文件名、需要读取的文件列表（绝对路径）
- 子代理用 `toolsets:["file"]` 限制工具集，减少 token 开销
- 模块文件中应包含完整的「子代理调用规格」（goal/context/所需文件/输出格式），这样主代理只需引用模块名
- **v9.0 新增**：审核子代理必须将完整报告写入文件并返回报告正文（不只是 PASS/FAIL）；修复子代理依赖此完整报告定位问题
- **占位符替换**：主代理在调用 delegate_task 前，必须将 context 模板中的占位符（`{书名}`、`{工作目录}`、`{章节编号}` 等）替换为实际项目路径值。各子代理模块的调用模板中的 `{...}` 均为待替换占位符

## 重要说明

**记忆系统使用方式：**
- **记忆系统**：采用 LLM 原生可读/可写的自然语言文档（故事圣经/人物弧线/剧情时间线/伏笔清单/角色数据库）
- **结构**：`记忆系统/bible/`（5个故事圣经文档：story_bible / character_arcs / plot_timeline / chekhovs_gun / character_database）+ `summaries/`（单章摘要）+ `phases/`（10章压缩块）
- **初始化**：项目初始化时自动创建记忆系统
- **自动维护**：每章写完后自动更新圣经 + 生成摘要
- **阶段压缩**：每10章自动将摘要压缩为梗概，确保跨会话记忆不丢失
- **跨会话恢复**：新会话读取圣经 + 压缩块即可秒回状态
- **手动管理**：功能18可查看/编辑/修复记忆系统

**强约束系统：**
- **默认启用**：每章写作后自动检查三个维度（金手指叙事/时间约束/叙事线监控）
- **叙事参谋**：不报违规，只在有风险时给出叙事建议
- **自动初始化**：约束文档不存在时自动创建，无需手动设置
- **手动触发**：`金手指` / `时间约束` / `叙事线` / `强约束`
- **存储位置**：`[书名]/记忆系统/constraints/` 下 3 个追踪文档

**叙事质量管理框架：**
- **写前预检自动触发**：「继续」/「下一章」命令在加载单章写作模块前自动执行写前状态核对
- **阻断与自愈**：缺字段阻断写作 → 自动回溯缺失来源（回读摘要/圣经/弧线）→ 修复记忆系统 → 重新预检通过 → 继续
- **规则集加载逻辑**：首次使用自动加载「通用默认」规则集；规则集文件缺失 → 降级为「通用默认」；若通用默认也缺失 → 使用内置五维默认参数（定义如下）作为最终兜底
- **存储位置**：YAML 链条随章节存档，每 10/50 章压缩至 phases/volumes/

**导出功能：**
- 网文格式：纯txt，适配各平台
- 电子书：epub格式，支持目录导航
- 投稿版：Word格式，标准投稿排版

## 🛑 操作确认点

| 类型 | 操作 | 说明 |
|------|------|------|
| **需确认** | 删除章节/设定文件 | agent 不得自主删除用户项目文件 |
| **需确认** | 覆盖记忆系统 | 覆写 bible/summaries 前必须告知 |
| **需确认** | 回滚版本 | 版本回退前展示差异，等用户确认 |
| **需确认** | 修改大纲/核心设定 | 级联影响大，展示改动摘要后确认 |
| **建议确认** | 新章节生成后（非连续写作模式） | 展示生成内容，用户点头再存档续写；连续写作模式（由「继续」/「下一章」自动触发、未被中断的多章连写）下跳过此确认，直接存档并续写下一章 |
| **建议确认** | 阶段压缩（每10章） | 摘要压缩为梗概前告知，压缩后展示 |
| **无需确认** | 单章生成 | 「继续」命令自动写入 |
| **无需确认** | 记忆系统初始化 | 首次创建项目时自动初始化 |
| **无需确认** | 强约束检查 | 每章后自动执行，仅在有风险时输出 |
| **无需确认** | 写前预检（自愈模式） | 缺字段自动阻断 → 回溯来源修复 → 重新预检，无需用户干预。**记忆系统缺失例外**：不阻断（见边界条件表） |
| **建议确认** | 规则集中途切换 | 切换后 YAML 链条第 3 区块字段变化，在 phase 结束点切换更安全 |
| **建议确认** | 导出章节 | 导出前展示章节范围和格式，等用户确认再写入文件 |

## ⚠️ 边界条件

| 场景 | 处理 |
|------|------|
| 模块文件缺失 | 加载指定模块失败时，跳过该模块功能并告知用户「模块文件缺失，功能暂不可用」，不中断整体流程 |
| 记忆系统缺失（自动修复） | 05_06 检测到记忆系统未初始化 → 写前预检记录缺失但不阻断写作 → 05_26 写后流水线前置检查自动调用 05_18-A 降级初始化，不告知用户、不询问。手动「预检」命令仅报告缺失，不自动修复 |
| 项目目录不存在 | 跨会话恢复找不到项目 → 提示用户项目可能已移动，转为场景A |
| 多个项目匹配 | 「继续」搜到多个 bible → 列出所有项目让用户选择 |
| 用户中断创作 | 已生成的章节文件保留在 `[书名]/正文/`，下次「继续」时从最后一章接着来 |
| 规则集文件缺失 | 所选规则集 .md 文件被删除 → 降级为「通用默认」，提醒用户规则集已失效 |
| 规则集中途切换 | 切换规则集后 YAML 链条第 3 区块字段变化 → 标记切换点，建议在当前 phase 结束后再切换 |
| 导出无章节 | 用户说「导出」但正文/目录为空 → 提示「暂无章节可导出」，不生成空文件 |
| 功能依赖未满足 | 项目未初始化时触发需要项目的功能（如爽点/大纲/导出） → 提示「请先写小说或继续已有项目」 |
| delegate_task 不可用 | **阻断流水线**，提示用户修复 delegate_task 配置后重试。写后流水线依赖独立子代理审核避免自审偏见，不可降级为主代理自审（与铁律「.done 标齐才允许写下一章」冲突） |

| **⚠️ 读者审稿已改为单章审核** | 05_25_功能模块.md 已更新为单章版审核流程（4维度评分，不跨章审阅）。字数检查归属05_06单章写作步骤4，读者审稿紧随字数检查之后触发。写后流水线使用 .done 文件锁保护（[书名]/.done/），每步完成写标记，跨会话自动恢复。 |
