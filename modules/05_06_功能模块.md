# 功能模块 6：单章写作

> **功能编号**：6
> **功能定位**：按四段式结构创作章节，控制节奏和字数，自动集成记忆系统维护长篇连贯性。

---

## 一、核心特性

| 特性 | 说明 |
|------|------|
| **四段式结构** | 开篇、铺垫、高潮、收尾 |
| **节奏控制** | 根据情节类型调整节奏 |
| **字数控制** | 根据立项定位单配置（默认2000-2500字/章，下限=要求值，上限=要求值+500） |
| **记忆系统集成** | 自动维护长篇连贯性 |

---

## 二、执行流程

### 阶段一：章节生成前（自适应上下文组装）

```
【执行步骤 — 自然语言记忆系统 v5.0】

0. 检查记忆系统状态
   - 确认 [书名]/记忆系统/ 目录是否存在
   - 如果不存在 → 跳过上下文组装，直接进入「阶段二」，使用降级方案
   - 如果存在 → 继续以下步骤

1. 【章节类型判定】根据章节规划表和当前进度，判定本章类型：

   | 类型 | 特征 | 上下文策略 |
   |------|------|-----------|
   | 🔥 高潮章 | 爽点释放/大战/关键转折 | 加配：+5 额外摘要 + 所有相关伏笔原始摘要 |
   | 📜 伏笔回收章 | 计划回收 ≥1 个远古伏笔 | 专项：回读伏笔首现章摘要 + 上次推进章摘要 |
   | 🌊 过渡章 | 日常/铺垫/支线展开 | 减配：核心上下文即可（见步骤 3 过渡模式） |
   | 📦 标准章 | 常规推进 | 默认配置 |

   判定依据：
   - 章节规划表标注的「章节类型」和「爽点类型」
   - chekhovs_gun.md 中状态为 🔄（准备回收）的伏笔
   - 上一章结尾的悬念强度

1.5. 【资源预加载 — 根据章节类型自动匹配参考资源】
   基于步骤1的判定结果，按以下硬性规则加载参考资源（不可跳过）：

   | 判定条件 | 强制加载资源 |
   |----------|-------------|
   | 章节类型 = 开篇冲突 | `references/黄金开篇技巧.md` |
   | 爽点类型字段 ≠ 空 | `references/爽文创作理论.md` + `references/爽点设计策略.md` |
   | 章节类型 = 高潮爆发/大高潮 且 冲突类型 = 战斗冲突 | `references/打斗场景创作指南.md` |

   多条规则可同时触发（如：开篇首战章同时匹配开篇+爽点+打斗）。
   加载的资源内容将在步骤3上下文组装时纳入写作指导。

2. 【上下文预算】组装前估算总量，确保聚焦：
   - 目标：上下文总量控制在 100KB 以内（1M 上下文 agent 可适当放宽）
   - 如果估算超限，按优先级裁剪：
     ① 砍远古 volume（保留最近 3 个）→ ② 减少额外摘要（5→3）→ ③ character_arcs.md 仅读活跃角色
   - 过渡章默认使用减配模式（见步骤 3），不触发预算检查

3. 【核心上下文组装】（RTCO 分级：P0 核心 → P1 重要 → P2 参考）

   P0 核心（所有章节必加载）：


   a) 故事圣经快照 [P0]
      → 读取 [P0] story_bible.md 的「当前状态快照」+「上一阶段梗概」

   P1 重要（标准章加载，过渡章减配）：

   b) 人物弧线（按权重 + 章节类型分级加载）[P1]

      【权重分级规则】
      - 主角级（96-100）：全量上下文（完整演化线 + 当前状态）
      - 核心配角（81-95）：当前状态 + 最近 10 章演化
      - 关键角色（61-80）：当前状态段落
      - 次要及以下（≤60）：仅姓名 + 状态摘要（1 句），除非本章为核心出场章节

      【章节类型调整】
      - 高潮章/伏笔回收章：核心配角升级为全量加载
      - 过渡章：核心配角降级为当前状态；关键及以下仅保留出场角色
      - 标准章：按权重默认规则

      【增长控制】
      - 如果 character_arcs.md 超过 30KB：
        * 将 500 章前的演化线压缩为每个角色一段「远古摘要」（3-5 句）
        * 原始细节下沉到对应 volume 压缩文件中，按需回读

   P2 参考：

   c) 剧情时间线（聚焦近期）[P2]
      → 读取 plot_timeline.md 的最近 20 章事件因果链
      → 检查是否有支线超过 50 章未推进（标记提醒用户）

   d) 伏笔清单 [P2]
      → 读取 chekhovs_gun.md 全部活跃伏笔
      → 筛选：状态为 🔄 准备回收 的 → 标记为本章优先处理
      → 筛选：回收窗口已过期（当前章 > 预期回收章 + 10）的 → 标记 ⚠️ 提醒用户

4. 【章节类型专属加载】

   🔥 高潮章加配：
      - 额外读取最近 10 章摘要（而非默认 5 章）
      - 读取所有涉及角色的完整弧线
      - 读取最近 2 个 volume 压缩（宏观历史回顾）

   📜 伏笔回收章专项加载：
      - 对每个计划回收的伏笔，读取其首现章节的原始摘要
      - 如果伏笔在多章中被推进，读取最近一次推进章节的摘要
      - 回顾首现章的场景氛围、人物情绪、对话细节
      - 确保第 500 章回收第 1 章伏笔时拥有完整语境

   🌊 过渡章减配：
      - 只读 story_bible 状态快照 + 最近 3 章摘要
      - character_arcs 仅读本章出场角色
      - 跳过 volume 加载
      - 跳过伏笔详细审查（仅确认无意外回收）

5. 【Volume 智能读取】
   - 默认读取最近 5 个 volume 压缩文件（排除 _template.md）
   - 如果伏笔涉及更早的 volume → 按需回读相关 volume
   - 如果 volume 总数超过 10 个 → 最近 5 个全读 + 其余仅在伏笔涉及时按需加载
   - 过渡章：仅读取最近 1 个 volume（宏观背景）

6. 【风格同步】
```

### 阶段二：章节生成

```
0. 【读取字数配置】
   - 读取 [书名]/设定/[书名]_立项定位单.md，提取「每章字数」字段
   - 如果读到 → 字下限=要求值，字上限=要求值+500
     * 例：用户要求4000字 → 范围 4000-4500
   - 如果未读到（旧项目无此字段）→ 默认 2000-2500
   - 将字数范围记入变量，供后续步骤使用
   - 同时检查 [书名]/记忆系统/bible/story_bible.md 的「每章字数」字段作为备用

1. 【加载蓝图】
   - 读取章节规划表中本章的「出场角色」「场景」「冲突目标」字段
   - 确认蓝图要求的角色/地点/冲突在上下文中可用
   - 如果蓝图缺失 → 基于上一章摘要自动推断本章目标

1. 确认章节信息

```
【执行步骤】

1. 确认章节信息
   - 章节编号
   - 章节名称
   - 预期字数：[字下限]-[字上限]字（根据步骤0从立项定位单读取的配置。不足下限必须扩写，超过上限必须精简）

2. 加载四段式结构模板
   - 模板文件：templates/单章结构模板.md
   - 结构：开篇(10-15%) → 铺垫(20-30%) → 高潮(40-50%) → 收尾(15-20%)

3. 生成章节正文
   - 基于记忆包的上下文
   - 参考大纲或章节规划
   - 按四段式结构创作
   - 控制字数在预期范围内

4. 风格模仿（如用户指定）
   - 如果用户要求模仿特定作家风格 → 读取 `references/风格技巧详解.md`
     * 若该作家在速查表中 → 直接应用其句长/爽点/金句特征
     * 若不在表中 → 用三步分析法（句长指纹→爽点公式→金句模式）自行拆解
   - 将风格特征融入本章创作
   - 对话密集：参考《对话写作技巧》
   - 动作场景：参考《打斗场景创作指南》
   - 情感描写：参考《情感描写技巧》

5. 保存章节正文
   - 文件格式：纯文本，第一行为章节标题，第二行起为纯正文
   - 文件路径：[书名]/正文/第XXX章_章节名.txt
   - 严禁包含元数据和Markdown格式标记

6. 【字数强制检查 — 保存后立即执行】
   - 用 read_file 读取刚保存的章节文件
   - 统计中文字数（不含标点空格，仅统计 CJK 字符）
   - 字数 < [字下限] → ❌ 不合格，必须扩写。分析哪一段可以展开描写（对话/场景/心理/动作），重新生成阶段二步骤3-5
   - 字数 > [字上限] → ❌ 不合格，必须精简。删除冗余描写和重复信息，重新生成阶段二步骤3-5
   - [字下限] ≤ 字数 ≤ [字上限] → ✅ 合格，继续步骤7
   - 此步骤不可跳过。不合格的章节不得进入读者审稿流程。

7. 【读者审稿 — 字数检查通过后立即触发】
   - 加载 `modules/05_25_读者审稿.md`
   - 对刚完成的章节执行读者审稿流程（步骤1-5）
   - 输出读者审稿报告
   - 此步骤每章必做，不可跳过。
```

### 阶段三：章节生成后

```
【执行步骤 — 自然语言记忆系统 v5.0】

1. 生成单章摘要（L2）
   - 按照 [书名]/记忆系统/summaries/_template.md 的格式
   - 写入 [书名]/记忆系统/summaries/ch_{章节编号}_summary.md
   - 包含：
     * 核心事件
     * 人物变化（情绪/关系/目标）
     * 新埋设伏笔（同时更新 chekhovs_gun.md）
     * 已回收伏笔（同时更新 chekhovs_gun.md）
     * 本章结尾悬念
     * 下一章方向建议

2. 更新故事圣经（L1 — story_bible.md）
   - 更新「当前状态快照」
   - 更新「当前已写章节」计数
   - 记录本章结尾场景和悬念

3. 更新人物弧线（L1 — character_arcs.md）
   - 对本章涉及的角色，追加演化记录：
     * 情绪状态变化
     * 与其他角色的关系变化
     * 动机/目标调整
   - 更新该角色的「当前状态」段落

4. 更新剧情时间线（L1 — plot_timeline.md）
   - 追加本章事件到主线时间线
   - 更新相关支线的状态
   - 更新关键物品/线索追踪
   - 执行时间线一致性检查

5. 更新伏笔清单（L1 — chekhovs_gun.md）
   - 新埋设的伏笔 → 添加到清单（状态：已埋设，记录首现章节）
   - 已回收的伏笔 → 标记为（状态：已回收），在近况中追加「第X章回收：[简述回收方式]」
   - 即将回收的伏笔 → 标记为（状态：准备回收）

   【必须执行】遍历所有活跃伏笔（状态为 🔘 或 🔄），逐条检查：
   a) 本章是否涉及该伏笔？（推进线索 / 部分揭露 / 状态变化 / 角色获得新信息）
   b) 如果是 → 立即更新近况字段，用 1-2 句自然语言记录本章进展
      格式：「第X章：[本章发生的推进内容]」
      追加到现有近况末尾，不覆盖历史
   c) 如果不是 → 跳过
   d) 对于状态为 🔄（准备回收）的伏笔，额外检查：
      - 本章是否完成了回收？如果是 → 标记为 ✅ 并记录
      - 回收窗口是否已过？（当前章节 > 预期回收章节 + 10）→ 标记 ⚠️ 过期，提示用户

   此步骤不可跳过。即使本章没有明显伏笔推进，也要执行遍历确认。

6. 阶段压缩检查
   - 如果本章编号是 10 的倍数（第10/20/30...章）：
     * 读取最近10个单章摘要
     * 按照 [书名]/记忆系统/phases/_template.md 格式
     * 压缩为 [书名]/记忆系统/phases/phase_{序号}_ch{起始}-{结束}.md
     * 同时更新 story_bible.md 中的「故事阶段划分」
   - 如果本章编号是 50 的倍数（第50/100/150...章）：
     * 复查所有高重要度伏笔的近况，确保无遗漏（此检查从步骤5移至此，与卷压缩一并执行）
     * 读取最近5个 phase 压缩文件
     * 按照 [书名]/记忆系统/volumes/_template.md 格式
     * 压缩为 [书名]/记忆系统/volumes/volume_{序号}_ch{起始}-{结束}.md
     * 这是全书最高层级的记忆缓存
   - 非整数倍：跳过

6.5. 周期性自动检查（与阶段压缩同步触发）

   **每 5 章（第5/10/15/20...章）：**
   - 加载 `modules/05_17_功能模块.md`（连贯性检查）：
     * 对照 character_arcs.md 检查人物一致性
     * 对照 plot_timeline.md 检查情节逻辑
     * 对照 chekhovs_gun.md 检查伏笔回收状态
   - 加载 `modules/05_13_功能模块.md`（质量检查）深度模式：
     * 人物一致性、情节逻辑、设定符合度、伏笔回收
   - 非整数倍：跳过

   **每 10 章（第10/20/30...章，与阶段压缩同步）：**
   - 加载 `modules/05_14_功能模块.md`（创新检查）：
     * 情节类型多样性、人物成长深度、主题体现、创新元素

7. 五道生成门禁（不通过不视为完成）

   【门禁一：引用校验】
   - 本章涉及的角色/地点/势力是否都在设计文档（人设卡/世界观设定）中存在？
   - 新登场的角色/地点/势力是否已登记到对应文档？
   - 违反 → 标记 ⚠️ 并提示用户补充设定

   【门禁二：一致性校验】
   - 对照 character_arcs.md：角色行为是否 OOC？实力/关系/位置是否矛盾？
   - 对照 story_bible.md：物品状态/势力格局是否冲突？
   - 违反 → 标记 🔴 并列出具体矛盾点，建议修正

   【门禁三：蓝图出场校验】
   - 对照章节规划表的「出场角色」和「冲突目标」字段
   - 蓝图指定的角色是否实际出场？冲突目标是否推进？
   - 超过 2 个指定角色未出场 → 标记 ⚠️ 提醒

   【门禁四：未知实体检测】
   - 统计本章新登场的人物/地点/势力数量
   - 区分两类：有剧情作用的角色（需登记）vs 一次性路人/场景（无需登记）
   - 需登记的新实体超过 5 个 → 标记 ⚠️ 警告：「本章引入过多新实体，建议拆分到多章或精简」
   - 所有需登记的实体 → 立即追加到对应设计文档（人设卡/世界观设定/势力表）
   - 此门禁防止 agent 失控引入大量未登记角色导致设定混乱

   【门禁五：描写一致性】
   - 对照人设卡：本章对已有角色的外貌描写（发色/瞳色/体型/标志特征）是否与设定冲突？
   - 对照世界观设定：本章对已知地点的描写是否与之前的设定矛盾？
   - 冲突 → 标记 🔴 并列出具体冲突：「第 X 章设定张三黑发，本章写金发」
   - 此门禁防止视觉/设定层面的低级矛盾

8. 状态偏离检测
   - 扫描 character_arcs.md：是否有角色实力倒退未解释？
   - 扫描 chekhovs_gun.md：是否有伏笔超过预期回收章节 10 章仍未回收？
   - 扫描 plot_timeline.md：是否有支线超过 50 章无推进？
   - 发现偏离 → 标记 ⚠️ 并在完成报告中列出

8. 风格与质量自检
   - 检查本章文风是否与前期章节一致
   - 如有问题 → 提示用户并提供修复建议
   - 如无问题 → 输出完成报告
```

## 三、正文文件格式规范

```
【文件名格式】
第XXX章_章节名.txt

【文件内容格式】
第一行：章节标题
第二行起：纯正文内容
```

**示例**：

```
第一章：觉醒

（正文内容开始...）
```

**注意事项**：
- **严禁**在文件开头添加元数据（如"本章字数：XXX字"）
- **严禁**使用Markdown格式标记（如#标题、**加粗**等）
- **严禁**在文件结尾添加额外说明
- 文件必须是纯文本格式，仅包含章节标题和正文内容

## 四、批量写章节流程

当用户请求写接下来N章时，循环执行单章写作流程N次：

```
for i in range(N):
    章节编号 = 当前章节 + i
    执行阶段一：章节生成前
    执行阶段二：章节生成
    执行阶段三：章节生成后
```

## 五、记忆系统集成说明（v5.0 自然语言记忆）

本模块集成了自然语言记忆系统的全部流程，在章节创作前后自动执行：

| 能力 | 实现方式 | 触发时机 |
|------|----------|----------|
| **L4 上下文包组装** | 直接读取 `[书名]/记忆系统/bible/` 4个文档 | 每章创作前 |
| **单章摘要回填** | 写入 `[书名]/记忆系统/summaries/ch_NNN_summary.md` | 每章创作后 |
| **圣经更新（4文档）** | 追加/修改 `bible/` 下自然语言文档 | 每章创作后 |

降级方案不生成单章摘要、阶段压缩和卷压缩。后续如需启用记忆系统，使用功能18（记忆系统管理）初始化。

## 九、注意事项

1. **文件格式**：严格按照正文文件格式规范保存章节文件
2. **字数控制**：字数范围从立项定位单读取（默认2000-2500）。不足下限必须扩写重来，超过上限必须精简重来
3. **记忆系统一致性**：每次写完必须更新至少4个圣经文档 + 1个摘要
4. **人物弧线**：参考 character_arcs.md，人物成长必须被记录和解释
5. **伏笔管理**：每次新埋伏笔必须同步到 chekhovs_gun.md
6. **阶段压缩不能省略**：第10/20/30...章必须执行压缩
7. **跨会话恢复**：新会话必须读取 bible/ 3个阶段 + 最新phase + 最近5章摘要
8. **问题处理**：如发现人物跑偏或剧情矛盾，立即标记并提示用户
