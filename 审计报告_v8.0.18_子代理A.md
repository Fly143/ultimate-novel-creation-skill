## 二、逻辑完整性审计

### 🔴 LG-01：FAIL 修复闭环不完整——修复子代理可能引入新问题而未被检测

**涉及文件**：`modules/05_26_功能模块.md` 第164-166行

**当前流程**：
```
任一 FAIL → 启独立修复子代理读取FAIL报告+正文 → 修复 → 重启对应子代理二次审核
```

**问题**：修复子代理修改正文后，只重启**对应**FAIL 的子代理。但正文修复可能引入新的问题——例如修复「连贯性」问题时改动了某段描写，可能引入新的「五道门禁」或「质量」问题。当前流程不会重新触发那些原先 PASS 的子代理。

**场景示例**：
1. A(五道门禁) PASS, C(读者审稿) PASS, D(连贯性) FAIL
2. 修复子代理修改正文修复连贯性问题
3. 只重启 D 二次审核 → D 通过
4. 但修改后的正文可能触发了 A 或 C 的新问题 → **未被检测**

**修复建议**：在步骤7中增加规则：「修复子代理修改正文后，**所有子代理(A/C/D/E/F/G)均需重新审核**（B 强约束始终 PASS 除外），不得仅重启 FAIL 子集。二次全量审核全部 PASS 后方可标 .done。」

### 🟡 LG-03：步骤2 主代理自检「立即写入 .done_logic_selfcheck」的原子性缺陷
