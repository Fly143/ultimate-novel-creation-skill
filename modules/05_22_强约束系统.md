### 22. 强约束系统

**功能**：叙事参谋，不替作者做决定。在每章写作后自动检查三个维度——金手指叙事健康度、时间流逝感与紧迫度、多条叙事线的读者预期——发现潜在风险时给出提醒，不报「违规」。

**定位**：预防性诊断。不同于功能 17（事后检查一致性），本系统是**事前提醒**——在问题还没形成之前就点出来。

**触发方式**：
- **自动**：每章写作后（功能 6 结尾），作为记忆系统更新流程的最后一步
- **手动**：`金手指` / `时间约束` / `叙事线` / `强约束` → 加载本模块并定位到对应子系统

**存储位置**：
- `[书名]/记忆系统/constraints/cheat_constraints.md` — 金手指叙事状态
- `[书名]/记忆系统/constraints/time_constraints.md` — 时间约束追踪
- `[书名]/记忆系统/constraints/narrative_threads.md` — 叙事线监控

## 子系统 1：金手指叙事约束

### 追踪什么

| 维度 | 检查内容 |
|------|---------|
| **代劳率** | 金手指是否在替主角解决问题？主角自身成长（智慧/毅力/人际关系）推动了什么？ |
| **代价感知** | 最近几章使用金手指时，读者看到代价/消耗/副作用了吗？ |
| **新鲜感** | 展示方式是否重复？最近几次都是同一种套路？ |
| **膨胀匹配** | 金手指的解决能力，和当前剧情压力对等吗？ |
| **来源闭环** | 近期的使用，还跟金手指的来源/设定自洽吗？ |

### 初始化

- 用户说 `金手指` → 加载本模块，从已有设定文档中提取金手指信息，创建追踪文档
- 初始化模板见 `templates/constraints/cheat_constraints_template.md`

### 每章写作后自动检查

```
1. 【读取】读取 [书名]/记忆系统/constraints/cheat_constraints.md
   - 如不存在 → 从已有设定文档中自动提取金手指信息，创建追踪文档
   - 不询问用户，默认初始化

2. 【分析本章】
   - 金手指在本章中使用了几次？解决了什么问题？
   - 最近 5 章中，金手指代劳 vs 主角自身成长的比例

3. 【更新记录】
   - 追加本章金手指使用记录
   - 更新「近期使用模式」摘要

4. 【输出】（仅在有风险时输出）
   - 叙事风险提示，不是违规报告
```

### 输出示例

```
⚠️ 金手指叙事提示 — 第 27 章后

• 代劳倾向：近 5 章中，主角靠金手指解决问题 7 次，靠自身智慧/毅力仅 2 次
  读者可能感觉「金手指才是主角」
→ 建议：接下来 2-3 章安排一次主角脱离金手指、靠自身能力解决的危机

• 零代价：金手指连续使用 6 次，读者没看到任何代价或消耗
  读者可能觉得「反正随便用，剧情没悬念」
→ 建议：下次使用时让读者感受到冷却/消耗/副作用

• 新鲜度衰减：最近 3 次使用都是「默念口诀→金光一闪→解决」
  套路化会让爽感打折
→ 建议：换个展现方式，或者让使用过程出一次意外
```

### 零风险时不输出

如果一切正常，静默通过，不在终端刷存在感。只在有值得提醒的点时才说话。

## 子系统 2：时间约束

### 追踪什么

| 维度 | 检查内容 |
|------|---------|
| **时间流逝感** | 最近几章过去了多久？读者感受和故事内时间一致吗？ |
| **紧迫度叙事** | 如果有 deadline，读者在最近的章节里还能感受到压力吗？ |
| **节奏匹配** | 大事前夕是否写了太多日常？还是赶太急跳过了该有的铺垫？ |

### 初始化/添加 deadline

- 大纲阶段或用户随时说「加个 deadline：30 天后宗门大比」
- 用户说 `时间约束` → 查看/管理时间约束

### 初始化模板

见 `templates/constraints/time_constraints_template.md`

### 每章写作后自动检查

```
1. 【读取】读取 [书名]/记忆系统/constraints/time_constraints.md
   - 如不存在 → 自动创建空追踪文档
   - 不询问用户

2. 【更新流逝】
   - 本章故事内过了多久（天/小时/月）
   - 累计总时间、距离各 deadline 的剩余时间
   - 追加到时间线记录

3. 【检查紧迫度】
   - 是否有 deadline 正在逼近？
   - 最近的章节是否在叙事上维持了这份紧迫感？

4. 【检查流逝比例】
   - 故事内时间流速和章节进展比例是否合理？
   - 200 章只过了 3 天 vs 3 章过了 10 年，分别有什么叙事影响？

5. 【生成提醒】(仅在有风险时输出)
```

### 输出示例

```
⚠️ 时间叙事提示 — 第 35 章后

• 宗门大比（第 20 章埋下，说「三个月后」）
  已过 15 章，故事内仅过 5 天
  读者的「deadline 思维」已经模糊
→ 建议：后续章节加快时间流速，或加入时间跳跃

• 紧迫度断裂：第 30-32 章还在渲染大比压力，第 33-35 章转入日常线
  压力线断了 3 章，读者可能已经放松
→ 建议：下一章用一个细节（如收到宗门催函）拉回紧迫感

• 节奏倒挂：主角「三天后突破」的紧迫节点，插了一整章咖啡厅闲聊
  读者的焦急预期被稀释
→ 回顾是否可以把闲聊压缩到半章
```

## 子系统 3：多条叙事线监控

### 追踪什么

| 维度 | 检查内容 |
|------|---------|
| **被遗忘的线** | 某条线超过 N 章没提，且它埋的悬念读者还在等 |
| **情绪节奏** | 所有活跃线最近都是同一情绪基调？读者需要喘息 |
| **预期负债** | 某条线卡在悬念处太久，读者的等待在贬值 |

### 初始化

- 大纲阶段或用户添加叙事线
- 用户说 `叙事线` → 查看/管理叙事线

### 初始化模板

见 `templates/constraints/narrative_threads_template.md`

### 每章写作后自动检查

```
1. 【读取】读取 [书名]/记忆系统/constraints/narrative_threads.md
   - 如不存在 → 从已有设定（大纲、人物关系图谱）自动提取叙事线信息，创建追踪文档
   - 不询问用户

2. 【更新状态】
   - 本章涉及了哪些叙事线
   - 更新各线的「最后活跃章节」

3. 【检查遗忘】
   - 某条线超过 10 章没人碰，且状态为「活跃」→ 提醒
   - 某条线停留在悬念高潮处超过 15 章 → 读者预期正在贬值

4. 【检查情绪单调】
   - 近 5 章所有活跃线是否全是同一情绪基调

5. 【生成提醒】(仅在有风险时输出)
```

### 输出示例

```
⚠️ 叙事线提示 — 第 42 章后

• 支线「苏晴的身世」：第 25 章揭了半个悬念后 17 章没碰
  读者还在等，但耐心在递减
→ 建议：3 章内切回去一次，哪怕只给一个细节/伏笔，保持悬念热度

• 情绪单调：主线 + 支线 1「宗门斗争」近 5 章全在压抑/紧张基调
  读者连续紧张太久会疲劳
→ 建议：接下来 1-2 章让某条线出现小爽点或温情戏，给读者呼吸空间

• 线索「林萧的玉佩」：第 15 章后未被任何线提及
  该线索属于主线设定，长期冷落可能导致后期回收时生硬
→ 建议：近期让某个角色不经意提及玉佩，做一次轻量存在感刷新
```

## 约束系统与记忆系统的关系

| | 记忆系统（bible/） | 约束系统（constraints/） |
|---|---|---|
| **用途** | 记录事实（发生了什么） | 分析风险（可能会发生什么） |
| **更新方式** | 每章追加 | 每章分析 |
| **性质** | 工具书 | 参谋 |
| **输出** | 不主动说话 | 有风险才提醒 |

约束系统**不替代**记忆系统，而是在记忆系统之上做一层叙事健康分析。

| 命令 | 执行 |
|------|------|
| `强约束` / `约束检查` | 执行全部三个子系统的检查 |
| `金手指` / `金手指约束` | 仅检查/管理金手指约束 |
| `时间约束` / `deadline` | 仅检查/管理时间约束 |
| `叙事线` / `支线监控` | 仅检查/管理叙事线监控 |

手动触发时，如果对应约束文档不存在，先执行初始化（从已有设定文档中提取信息），再执行检查。
