# 功能模块 26：写后流水线（强制·每章必跑·v9.1）
>
> **功能编号**：26
> **功能定位**：单章正文通过字数检查后，执行"主代理记忆维护 + 子代理审核 + 子代理修复"的完整闭环。
> **架构版本**：v9.1 — 主代理从"写手"转为"编辑"：只负责记忆维护（步骤2）+ 调度审核/修复子代理。写作（05_06）、审核、修复全部由独立子代理完成。
> **铁律**：第 N 章全部 PASS + .done 标齐，**才允许写第 N+1 章**。黄金开篇"第1-3章"也必须逐章闭环，禁止连写再补审。

> 写后流水线依赖记忆系统（bible/summaries/phases）的正常运转。如果记忆系统未初始化，流水线的步骤2（圣经更新）和所有子代理将无法读取所需文件。

**执行逻辑**：
1. 检查 `[书名]/记忆系统/bible/` 目录下的 5 个圣经文档是否全部存在：
   - story_bible.md / character_arcs.md / plot_timeline.md / chekhovs_gun.md / character_database.md
2. **全部存在** → 记忆系统已初始化，正常进入步骤1
3. **任一缺失** → 标记为**降级模式**：
   - 执行 05_18 工作流 A（项目初始化），从模板创建记忆系统目录结构
   - 用当前已有设定文档（大纲/人设/世界观）填充圣经
   - 初始化完成后重新进入步骤1
   - ⚠️ 降级模式下：步骤2 只能创建首章摘要和初始圣经（无前文可对比），子代理 C/D/E 的连贯性检查降级为"无前文基线，仅查本章自洽"

## 一、总流程（顺序不可跳）

```
步骤1  字数确认（复用 05_06 步骤4 脚本结果）→ 标记 .done_wordcount
步骤2  主代理更新 5 圣经 + 摘要（编辑职责·唯一保留的动手环节）
步骤3  第一批子代理（3 并行）：五道门禁 A / 强约束 B / 逻辑因果审查 G（合并原步骤2+原G，共7项）
步骤4  第二批子代理（3 并行）：读者审稿 C / 连贯性 D / 节奏 E
步骤5  第三批子代理（1）：质量 F
步骤6  收集结论 → 全 PASS 标 .done；有 FAIL → 独立修复子代理（只读审核报告+正文，不读记忆文档）
步骤7  周期检查（每5/10/50章触发）
步骤8  追加章节元数据
```

> **v9.0 架构变更说明**：
> - **原步骤2 主代理逻辑自检已移除**：v9.0 起写作由独立子代理完成，主代理不再持有"写手独有视角"（设定全貌+写作意图），原步骤2 的 4 项检查（设定自打架/信息越级/前文蓝图一致性/伏笔对齐）已合并至步骤3 子代理 G（扩展至 6 项）。
> - **审核子代理改为返回完整报告**：不再只返回 PASS/FAIL 一行。完整审核报告使得修复子代理可以直接读报告定位问题，无需重新加载全套记忆文档。
> - **修复子代理只读审核报告+正文**：不再需要读取记忆文档。审核报告已包含"具体段落引用→矛盾点→修改建议"，修复子代理按报告指示执行即可。

### 步骤2 详细：主代理更新 5 圣经 + 摘要（★权威定义，05_06 步骤5→05_26 步骤2 唯一权威定义）

> 本步骤是主代理唯一保留的动手环节（编辑职责）。主代理阅读写作子代理产出的正文，更新记忆系统。

1. 生成单章摘要（L2）
   - 按照 `[书名]/记忆系统/summaries/_template.md` 的格式
   - 写入 `[书名]/记忆系统/summaries/ch_{N}_summary.md`
   - 包含：核心事件 / 人物变化（情绪/关系/目标）/ 新埋设伏笔（同步 chekhovs_gun.md）/ 已回收伏笔（同步 chekhovs_gun.md）/ 本章结尾悬念 / 下一章方向建议
2. 更新故事圣经（story_bible.md）：当前状态快照（结尾场景/人物在场/最后一句话/主角目标/主要障碍/读者悬念）、当前已写章节计数
3. 更新人物弧线（character_arcs.md）：本章涉及角色的演化记录（情绪/关系/目标变化）+ 当前状态段落
   - 同步更新角色数据库（character_database.md）：出场/退场事件、能力状态变化、物品持有变动、关系快照
4. 更新剧情时间线（plot_timeline.md）：
   - 在主线时间线表格中追加本章事件，**必须加标签**（[主线]/[支线X]/[交叉]）
   - 更新活跃支线状态表（上次活跃章节/当前状态/核心悬念）
   - 更新支线详情中的事件时间线
   - 关键物品/线索有变动时更新追踪表
5. 更新伏笔清单（chekhovs_gun.md）：新埋设→添加（状态🔘已埋设，记首现章）；已回收→标记✅并追近况；🔄准备回收→检查本章是否回收/是否过期（>预期+10章标⚠️）

## 三、步骤6 结论收集与修复（★v9.1 重构）

### 6.1 结论收集与分类

等待步骤3/4/5 全部子代理返回（A/B/C/D/E/F/G 共 7 个结论）。

- ⚠️ **B（强约束）始终 PASS**：不触发修复，不阻断流水线。
- 其余六项（A/C/D/E/F/G）按三态分类：

| 状态 | 含义 | 操作 |
|------|------|------|
| `PASS` | 无任何问题 | 标对应 .done |
| `PASS（含优化建议）：[N]条` | 有可优化点但非逻辑硬伤（如词汇重复、氛围渲染不足、配角单薄等） | 标对应 .done + 主代理记录建议入报告摘要，**不触发修复子代理** |
| `FAIL: [N]个问题` | 存在逻辑硬伤（因果断裂/设定矛盾/OOC等） | **触发修复子代理** |

### 6.2 全 PASS / PASS（建议）路径

六项全部为 `PASS` 或 `PASS（含优化建议）`、无任何 `FAIL` → 标 `.done_chapter`：

- 逐标记 `.done_gates` / `.done_constraints` / `.done_logic_causal` / `.done_review` / `.done_coherence` / `.done_rhythm` / `.done_quality`
- 如有 `PASS（含优化建议）`项 → 主代理收集所有建议，在流水线总结中汇总输出，但不阻断闭环
- 全部齐备后标 `.done_chapter`
