### 25. 围棋对局模块（Go Game Integration）

**功能**：为围棋题材小说提供真实对局谱支持。当小说写到围棋对局场景时，自动从内置棋谱库检索匹配的真实对局，将 SGF 棋步转换为自然语言描写。

**核心原则**：所有棋步必须来自真实职业对局，禁止 LLM 自行编造围棋着法。

**关于棋谱解说（C[] 注释）**：
- 有解说的棋谱（如名局精解 5 局）→ 可以直接引用吴清源原话，小说更厚重
- **无解说的棋谱同样可用** → LLM 的围棋知识足以理解每步棋的战略意图（分投、腾挪、弃子、收官等），只要有真实坐标就能写出合理的局面描写
- 棋谱库中 1,160 局吴清源全集（flygo.net）虽无详细解说，但坐标完全真实，**全部可用**。不必局限于名局精解那 5 局

**触发方式**：
- 手动：`围棋对局` / `写一局棋` / `对局` / `棋谱`
- 自动：在「继续创作」流程中，检测到当前章节出现围棋对局描写时，必须暂停写作，加载本模块完成棋谱检索后再续写

## ⚠️ 强制协议

**当小说写到围棋对局场景时，你必须执行以下流程，不得跳过：**

1. 停止对局面的 LLM 自由生成
2. 从棋谱库中检索真实对局
3. **先扒谱：从 SGF 中提取真实的坐标表（🔴 必须先做这步）**
4. **基于坐标表写对局描写（每步必须对应真实坐标）**
5. 完成后回到主线续写

**任何自行编造的围棋着法都可能导致棋理错误，破坏小说专业性。**

## 🔴 铁律：一局棋 = 一个 SGF 文件

**同一场对局中的所有着法必须来自同一个 SGF 文件，不得混用。** 

- 选定了棋谱文件后，从头到尾只读取这一个文件的手数
- 绝不允许：开局用 A 棋谱的第 1-30 手、中盘用 B 棋谱的第 50-100 手
- 每个对局场景必须对应且仅对应一个 SGF

**如何确保：**
- 选定棋谱后，用 `read_file` 一次性读取完整的 SGF 内容
- 从第一手读到最后一手，确保每一步在这个文件中都是连续的
- 如果只需要棋谱的局部（例如只要布局的前 50 手），记得这只是裁剪而非拼凑
- 如果小说需要多个对局片段（第一章一盘、第五章一盘），各用各的 SGF

## 🔴🔴 核心改进：写棋前必须先扒谱（提取真实坐标表）

**这是本模块最重要的改动。** 以前的问题是：agent 读了 SGF 文件后直接凭记忆下笔，结果编造了棋步。现在的要求是：**先扒出真实坐标表，然后对着表写。每一步都必须能在这张表里找到对应。**

### 扒谱步骤（纯自然语言操作）

选定 SGF 文件后，执行以下步骤：

**步骤 A — 读文件**
用 `read_file` 完整读取 SGF 文件。

**步骤 B — 确定对局信息**
从文件头部提取：
```
PB[黑方姓名] PW[白方姓名] RE[结果] DT[日期]
```

**步骤 C — 提取主分支坐标（🔴 关键！）**

SGF 文件的结构：
- 根节点：`(;属性1[值]属性2[值]...`
- 着法节点：`;B[pd]`（黑棋落在 pd）、`;W[dd]`（白棋落在 dd）
- 变招：`( 分支A )( 分支B )`，括号包围的是分支

**🔴 SGF 解析注意事项（血泪经验）：**

SGF 文件可能包含 `\r\n` 换行符，且整个文件被包裹在最外层 `()` 内。直接用正则提取 `;B[xx]`/`;W[xx]` 而不处理括号深度，会把变招分支的着法也提取进来，导致坐标混乱。

**必须用括号深度跟踪法解析。** 方法：

1. 从文件开头逐字符扫描
2. 遇到 `(` → 深度+1，进入分支
3. 遇到 `)` → 深度-1，退出分支
4. 遇到 `;B[xx]` 或 `;W[xx]` 且深度=1 → 记录为主分支着法
5. 遇到 `;B[xx]` 或 `;W[xx]` 且深度>1 → 变招分支，**只取第一个子分支**（即第一个 `(` 后面的着法），跳过其他分支

**关键规则：**
- 第一个 `(` 之后的第一组着法是实战主分支
- 第二个 `(` 开始的是参考图/变招，跳过
- 如果第一个 `(` 里面还有嵌套的 `(`，同样只取第一个子分支

**提取后，把坐标表写在草稿中供自己逐手对照：**

```
===== 坐标表（主分支）=====
第1手 B[qd]
第2手 W[dc]
...
第26手 W[er]    ← 关键一手！
...
===== 坐标表结束 =====
```

**步骤 D — 转换为位置描述**

坐标 → 位置的对照规则：

**🔴 坐标系统说明（重要！）：**
SGF 坐标 `ab` 中，**第一个字母是列（a=左，s=右），第二个字母是行（a=上，s=下）**。
- 列：a=第1列(最左), s=第19列(最右)
- 行：a=第1行(最上), s=第19行(最下)

**常见位置对照表：**

| SGF坐标 | 中文位置 |
|---------|---------|
| dd/pd/dp/pp | 左上/右上/左下/右下角星位 |
| jj | 天元 |
| jd/dj/pj/jp | 左/下/上/右边星位 |
| cd/dc | 左上小目 |
| qd/pq | 右上小目 |
| cp/pc | 左下小目 |
| qp/pq | 右下小目 |
| ce/ec | 左上高目 |
| qe/eq | 右上高目 |
| cf/fc | 左上目外 |
| qf/fq | 右上目外 |
| de/ed | 左上五五 |
| qe/eq | 右上五五 |

**通用转换规则：**
- 列 <= 6 → 左，>= 14 → 右，其他 → 中
- 行 <= 6 → 上，>= 14 → 下，其他 → （空）
- 到棋盘边缘距离 = 线数（1=一路，2=二路，3=三路）
- 组合：左上方三路、右下二路、中下一路……

**🔴 坐标转换验证（血泪经验）：**

转换完一定要用 `execute_code` 跑验证函数，不要凭感觉写。这次踩的坑：
- `er` → 列5(e), 行18(r) → 下方2路（不是"左边"！）
- `cm` → 列3(c), 行13(m) → 左边中腹（不是"下边"！）
- `cl` → 列3(c), 行12(l) → 左边中腹
- `bn` → 列2(b), 行14(n) → 左边下方

验证方法：用 `execute_code` 跑一个简单的坐标转换函数，输入坐标，输出中文位置描述，确认方向正确再动笔。

## 工作流程（agent 操作步骤）

### 场景 A：小说写到围棋对局时（自动触发）

**步骤 0 — 停止自由生成**
立即停止对局面的 LLM 自由生成。本模块加载后的所有围棋描写都必须基于真实数据。

**步骤 1 — 确认对局需求**
和用户确认以下信息：
- 这局棋是谁下的？（主角执黑还是白？对手是谁？）
- 需要什么类型的对局？（绝境逆转 / 碾压 / 半目险胜 / 肉搏杀棋 / 布局名局）
- 背景年代？
- 需要写多少手？

**步骤 2 — 检索棋谱**
用 `search_files(target='files', pattern='*.sgf', path='data/sgf/')` 列出可用棋谱目录。

根据条件挑选：
- 要「绝境逆转」-> 从番棋战或十番棋年代的棋谱中选一局 RE=白胜少目/B+R 接近的
- 要「碾压」-> 选 RE=B+R/W+R 的中盘胜局
- 要「布局名局」-> 优先从 `mingju_jingjie/` 选
- 指定了对手名 -> 用 `search_files` 在 SGF 文件内容中搜索对手名

**步骤 3 — 🔴 扒谱：提取真实坐标表（不可跳过！）**

用 `read_file` 读取 SGF 文件，然后按「核心改进」中的步骤 A-C 的方法，手动扒出主分支的真实坐标表。

**把扒出的坐标表写在草稿中供自己逐手对照**。格式如：

```
===== 坐标表（主分支）=====
第1手 B[pd] - 右上角星位
第2手 W[dc] - 左上小目
第3手 B[qp] - 右上三路
第4手 W[dq] - 左上三路
...
第26手 W[er] - 左上二路
...
===== 坐标表结束 =====
```

坐标表的数量就是你能写的步数上限。如果小说只需要前 50 手，只扒前 50 手即可。

**步骤 4 — 翻译为小说描写**

逐手对照 step-3 的坐标表。基本格式：
```
第{n}手，{黑/白}落于{坐标描述}。
```

**每写一步都必须回头看一眼坐标表中对应的真实坐标。** 当你写下「白26分投右下角」时，必须确认坐标表中第26手确实是右下角区域——如果不是就不能写，必须写坐标表中的真实位置。

大量着法可以简化合并：
```
布局阶段，黑1、3、5分别占据了右上星位、左下小目和天元。
白2、4、6则以二连星应对。
```

有 C[] 注释的直接引用：
```
{注释原文}
```

**步骤 5 — 结合叙事功能润色**

**步骤 6 — 写后自检（复核而非补救）**

因为 step-3 已经先扒了真实坐标表，自检只是复核。从正文中检出所有提到具体手数的围棋着法（至少 3 条），和 step-3 的坐标表交叉比对：

```
正文描写                    坐标表中的真实数据
「黑棋右上角星位」          -> 第1手 B[pd] ✅
「白12守角」                -> 第12手 W[ce] ✅
「白26」                    -> 第26手 W[er] ✅
```

**判定**：
- 全部正确 -> 通过
- 任意错误 -> 打回，从步骤 4 重新翻译（坐标表本身也可能有扒错的情况，如果是表错了就回步骤 3）

**步骤 7 — 章节末尾标注**
自检通过后，在本章末尾追加：
```
> 本章对局参考：[SGF文件名] | [黑方名] vs [白方名]([年份]) | 共[N]手
```

**步骤 8 — 回到主线写作**

### 实战对比：正确方法 vs 错误方法

**错误做法（之前出问题的原因）**：
1. read_file 读 SGF -> 扫一眼心里有个大概印象
2. 凭记忆下笔：「白26分投在右下角星位外侧」
3. 写完后搞个自检表：「白26 -> W[qk] ✅」
4. 实际白26是 W[er]，完全对不上
5. **根源问题是：写的时候就没有用真实坐标**

**正确做法（本模块要求）**：
1. read_file 读 SGF -> 用 Python 解析代码扒出真实坐标表
2. 确认第26手是 W[er]（下方二路），不是右下角
3. 用 execute_code 验证坐标转换：`er` → 列5行18 → 下方2路
4. 对着表下笔：「白26在下方二路长出一子……」
5. 每步都回头确认坐标表
6. 自检时因为写的时候用的就是真实坐标，自然能对上
7. **根源解决：写前就已经有真实坐标约束**

**本次重写踩到的坑（2026-05-15）**：

坑1：SGF 解析反复失败
- 原因：SGF 文件有 `\r\n` 换行，且整个文件被 `()` 包裹，直接用 `re.findall(r';B[xx]')` 提取会把变招分支也提取进来
- 解决：用递归括号深度跟踪法，只取第一个子分支（实战分支）

坑2：坐标转换方向写错
- 原因：凭感觉把 `er` 翻译成"左边"，实际 `e`=列5(中偏左)，`r`=行18(下方)，应该是"下方"
- 解决：用 execute_code 跑验证函数，每个坐标都验证一遍

坑3：棋步描写与坐标不符
- 原因：写了"白28拆二"、"黑29飞出"、"白30压"，实际 SGF 中这些棋步全在左边纠缠，没有拆二飞出压
- 解决：严格对照坐标表，`cl`=左边8路是"贴"不是"拆二"，`bm`=左边7路是"长"不是"飞出"

坑4：字数不足
- 原因：第一版只写1840字，不够2000字下限
- 解决：扩写训练室环境描写、老人呼吸细节、周清河收棋动作、林弈敲棋盘细节

## 注意事项

1. **永远不要自己编造围棋着法。** 每一手都必须来自真实的 SGF 棋谱的坐标表。
2. **可以裁剪棋谱长度。** 只需要前 50 手就只扒前 50 手。
3. **可以组合使用。** 写多个对局时每局各选一个真实 SGF。
4. **名局精解的注释可直接引用。**
5. **检索不到匹配棋谱时**告知用户，建议更换需求。
6. **坐标表就是你的底线约束。** 如果写了坐标表上没有的着法，那就是违规。
7. **🔴 SGF 解析必须用括号深度跟踪法。** 不要用简单的正则提取 `;B[xx]`/`;W[xx]`，会把变招分支的着法也提取进来。必须用递归解析，只取第一个子分支（实战分支）。直接用上面提供的 Python 代码。
8. **🔴 坐标转换必须用代码验证。** 不要凭感觉把 `er` 翻译成"左边"——`er` 是列5行18，实际是"下方2路"。每次转换完用 `execute_code` 跑一遍验证函数。
9. **🔴 棋步描写可以概括，但方向必须对。** 比如"双方在左边展开接触战"是可以的，但如果实际棋步在左边而写成了"下边"，就是错误。概括性描写的区域范围必须与坐标一致。
10. **字数不足时优先扩写环境描写和人物心理。** 不要为了凑字数而增加不存在的棋步。可以在训练室环境、人物表情、心理活动、棋局氛围等方面扩充。
11. **skill_view 加载问题。** 技能名称中的特殊字符（·）可能导致 `skill_view` 加载失败。解决方案：用 `skills_list()` 确认技能存在 → `find ~/.hermes/skills -name "SKILL.md" -path "*围棋*"` 定位文件 → `read_file` 直接读取。本技能实际路径：`~/.hermes/skills/go-novel-skill/SKILL.md`。
