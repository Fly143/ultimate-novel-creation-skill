## 📂 文件路径约定

**本 skill 中所有文件引用遵循以下规则：**

1. **skill 内部文件**：`modules/`、`references/`、`templates/`、`memory-system/`（skill 内模板）等路径，均相对于本 skill 的根目录。
   - 示例：`references/书名生成策略.md` → skill 目录下的 references/书名生成策略.md
   - 示例：`记忆系统/bible/story_bible.md`（不带 `[书名]/` 前缀）→ skill 内模板文件，只读

2. **项目文件**：带 `[书名]/` 前缀的路径，相对于当前工作目录。`[书名]` 是项目初始化时用户确认的书名。
   - 示例：`[书名]/记忆系统/bible/story_bible.md` → 项目目录下的记忆系统副本
   - 示例：`[书名]/设定/[书名]_人设卡.md` → 项目目录下的设定文件

3. **记忆系统文件区分**：
   - `memory-system/xxx`（无前缀）= skill 内的只读模板 → 仅用于项目初始化时复制
   - `[书名]/记忆系统/xxx`（有前缀）= 项目目录下的读写副本 → 所有后续操作的目标

4. **项目查找**：跨会话恢复时，在当前工作目录下搜索 `*/记忆系统/bible/story_bible.md` 定位已有项目。

## ⚠️ 自审悖论设计原则

> **核心教训**：同一个模型、同一个对话里"自己写自己审"是无效的。
> 审核结果天然偏向"还行"——因为审核器和写手共享上下文，已经被"洗脑"了。

**已实施的解决方案（v8.0.1+）**：

1. **结构分离**：所有审核任务（五道门禁/强约束/读者审稿/连贯性/节奏/质量/逻辑因果审查）通过 `delegate_task` 启动独立子代理执行
2. **上下文隔离**：子代理只读磁盘文件，不共享主代理的写作上下文
3. **结果隔离**：子代理只返回 `PASS` 或 `FAIL`，完整报告写入文件。主代理不读报告内容
4. **修复分离**：有 FAIL 时，启动独立修复子代理读报告并修复，而非主代理自己修
5. **字数统计脚本化**：用 Python 脚本统计 CJK 字符数，不让 LLM 数数（又慢又不准）

**不能脚本化的 vs 能脚本化的**：
- ❌ 伏笔判断、逻辑矛盾、描写充分度、节奏评估 → 必须靠模型（所以用独立子代理）
- ✅ 人名/地名/专有名词与设定列表的字符串匹配 → 可以脚本化（但目前未实现，优先级低）

**为什么"换人格"没用**：
- 给审核代理换个"毒舌编辑"语气 → 本质上还是同一个模型同一个上下文
- 人格设定只是表面变化，不解决上下文污染的根本问题
- 结构分离才是正解

**delegate_task 的边界**：
- ✅ 子代理的审核判断是独立的（没有写作偏见）
- ⚠️ 审核结果返回后，主代理的对话里同时有写作上下文+审核结果
- ⚠️ 修复决策还是主代理做的（但至少有了独立的外部输入）
- 完美方案需要审核和修复也分开（已实施：修复由独立子代理执行）

**Pitfall**：子代理 `max_concurrent_children` 限制（默认3），不能一次起超过3个并行子代理。超过时需分批（先3个，完成后再3个）。

**delegate_task 实操要点**：
- 子代理的 `context` 必须包含：项目工作目录路径、章节文件名、需要读取的文件列表
- 子代理用 `toolsets: ["file"]` 限制工具集，减少 token 开销
- 模块文件中应包含完整的「子代理调用规格」（goal/context/所需文件/输出格式），这样主代理只需引用模块名
- 子代理必须将报告写入文件（而非仅返回文本），确保主代理不接触审核细节

## 重要说明

**记忆系统使用方式：**
- **记忆系统**：采用 LLM 原生可读/可写的自然语言文档（故事圣经/人物弧线/剧情时间线/伏笔清单）
- **结构**：`记忆系统/bible/`（4个故事圣经文档）+ `summaries/`（单章摘要）+ `phases/`（10章压缩块）
- **初始化**：项目初始化时自动创建记忆系统
- **自动维护**：每章写完后自动更新圣经 + 生成摘要
- **阶段压缩**：每10章自动将摘要压缩为梗概，确保跨会话记忆不丢失
- **跨会话恢复**：新会话读取圣经 + 压缩块即可秒回状态
- **手动管理**：功能18可查看/编辑/修复记忆系统

**强约束系统：**
- **默认启用**：每章写作后自动检查三个维度（金手指叙事/时间约束/叙事线监控）
- **叙事参谋**：不报违规，只在有风险时给出叙事建议
- **自动初始化**：约束文档不存在时自动创建，无需手动设置
- **手动触发**：`金手指` / `时间约束` / `叙事线` / `强约束`
- **存储位置**：`[书名]/记忆系统/constraints/` 下 3 个追踪文档

**叙事质量管理框架：**
- **写前预检自动触发**：「继续」/「下一章」命令在加载单章写作模块前自动执行写前状态核对
- **阻断与自愈**：缺字段阻断写作 → 自动回溯缺失来源（回读摘要/圣经/弧线）→ 修复记忆系统 → 重新预检通过 → 继续
- **规则集默认**：首次使用自动加载「通用默认」规则集，可通过 `规则集` 切换
- **存储位置**：YAML 链条随章节存档，每 10/50 章压缩至 phases/volumes/

**导出功能：**
- 网文格式：纯txt，适配各平台
- 电子书：epub格式，支持目录导航
- 投稿版：Word格式，标准投稿排版

| **无需确认** | 记忆系统初始化 | 首次创建项目时自动初始化 |

## ⚠️ 边界条件

| 场景 | 处理 |
|------|------|
| 模块文件缺失 | 加载指定模块失败时，跳过该模块功能并告知用户「模块文件缺失，功能暂不可用」，不中断整体流程 |
| 项目目录不存在 | 跨会话恢复找不到项目 → 提示用户项目可能已移动，转为场景A |
| 多个项目匹配 | 「继续」搜到多个 bible → 列出所有项目让用户选择 |
| 用户中断创作 | 已生成的章节文件保留在 `[书名]/正文/`，下次「继续」时从最后一章接着来 |
| 规则集文件缺失 | 所选规则集 .md 文件被删除 → 降级为「通用默认」，提醒用户规则集已失效 |
| 规则集中途切换 | 切换规则集后 YAML 链条第 3 区块字段变化 → 标记切换点，建议在当前 phase 结束后再切换 |
| 导出无章节 | 用户说「导出」但正文/目录为空 → 提示「暂无章节可导出」，不生成空文件 |
| 功能依赖未满足 | 项目未初始化时触发需要项目的功能（如爽点/大纲/导出） → 提示「请先写小说或继续已有项目」 |
