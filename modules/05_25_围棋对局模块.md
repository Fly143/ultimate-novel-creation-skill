### 25. 围棋对局模块（Go Game Integration）

**功能**：为围棋题材小说提供真实对局谱支持。当小说写到围棋对局场景时，自动从内置棋谱库检索匹配的真实对局，将 SGF 棋步转换为自然语言描写。

**核心原则**：所有棋步必须来自真实职业对局，禁止 LLM 自行编造围棋着法。

**触发方式**：
- 手动：`围棋对局` / `写一局棋` / `对局` / `棋谱`
- 自动：在「继续创作」流程中，检测到当前章节出现围棋对局描写时，必须暂停写作，加载本模块完成棋谱检索后再续写

---

## ⚠️ 强制协议

**当小说写到围棋对局场景时，你必须执行以下流程，不得跳过：**

1. 停止对局面的 LLM 自由生成
2. 从棋谱库中检索真实对局
3. 基于真实棋谱生成对局描写
4. 完成后回到主线续写

**任何自行编造的围棋着法都可能导致棋理错误，破坏小说专业性。**

---

## 🔴 铁律：一局棋 = 一个 SGF 文件

**同一场对局中的所有着法必须来自同一个 SGF 文件，不得混用。** 

- 选定了棋谱文件后，从头到尾只读取这一个文件的手数
- 绝不允许：开局用 A 棋谱的第 1-30 手、中盘用 B 棋谱的第 50-100 手
- 每个对局场景必须对应且仅对应一个 SGF

**如何确保：**
- 选定棋谱后，用 `read_file` 一次性读取完整的 SGF 内容
- 从 `;B[pd]` 第一手读到最后一手，确保每一步在这个文件中都是连续的
- 如果只需要棋谱的局部（例如只要布局的前 50 手），记得这只是裁剪而非拼凑
- 如果小说需要多个对局片段（第一章一盘、第五章一盘），各用各的 SGF

## 工作流程（agent 操作步骤）

### 场景 A：小说写到围棋对局时（自动触发）

**步骤 1 — 确认对局需求**
和用户确认以下信息：
- 这局棋是谁下的？（主角执黑还是白？对手是谁？）
- 需要什么类型的对局？（绝境逆转 / 碾压 / 半目险胜 / 肉搏杀棋 / 布局名局）
- 背景年代？（如果是历史类小说则重要，现代背景可忽略）
- 需要写多少手？（整局还是只需关键片段？）

**步骤 2 — 检索棋谱**
用 `search_files(target='files', pattern='*.sgf', path='data/sgf/')` 列出可用棋谱目录。

然后根据条件挑选合适的对局：
- 要「绝境逆转」→ 从番棋战或十番棋年代的棋谱中选一局 RE=白胜少目/B+R 接近的
- 要「碾压」→ 选 RE=B+R/W+R 的中盘胜局
- 要「布局名局」→ 优先从 `mingju_jingjie/` 选
- 指定了对手名 → 用 `search_files` 在 SGF 文件内容中搜索对手名

**步骤 3 — 读取棋谱**
```python
# 用 read_file 读取 SGF 文件头判断对局信息
# SGF 头部包含：
# PB[黑方姓名] PW[白方姓名] RE[结果] DT[日期]
# 例如：PB[吴清源] PW[木谷实] RE[B+R] DT[1939-09-30]

# 然后读取全部手数：
# ;B[pd]  = 黑棋落在 pd 位置
# ;W[dd]  = 白棋落在 dd 位置
# ;B[jj]C[天元！] = 黑棋落在天元，有注释
```

**步骤 4 — 翻译为小说描写**
将 SGF 棋步转换为自然语言：

基本格式：
```
第{n}手，{执黑/白}落于{坐标描述}。
```

坐标 → 位置描述对照：
- `pd` (4-4) → 右上角星位
- `dd` (4-4) → 左上角星位
- `pp` (4-4) → 右下角星位
- `dp` (4-4) → 左下角星位
- `jj` (10-10) → 天元
- `cd` (3-4) → 左上角小目
- `pc` (16-3) → 右上角小目
- `qd` (17-4) → 右上角星位外侧
- `nc` (14-3) → 右边星位下两路

大量着法简化写法：
```
布局阶段，黑 1、3、5 分别占据了右上星位、左下小目和天元，
形成独特的"三三·星·天元"开局。
白 2、4、6 则以二连星应对。
```

如果 SGF 中有 C[] 注释（尤其 mingju_jingjie 中的），直接引用：
```
{注释原文}
```

**步骤 5 — 结合叙事功能润色**
根据用户想要的叙事效果，调整描写语气：
- 逆转：渲染前半盘的劣势 → 转折手的戏剧性 → 后半盘的步步紧逼
- 碾压：强调压倒性的气势 → 对手的无力感
- 险胜：渲染官子阶段的紧张 → 数子/半目的窒息感

**步骤 5.5 — 写后坐标自检（必须执行，内部操作，不写入正文）**

完成围棋对局段落的写作后，必须执行以下自检，不得跳过：

5.5.1 【列对照表】从本章正文中找出所有围棋着法描写，逐一标注对应的 SGF 坐标：

```
正文描写                          → SGF对应
"黑棋落在右上角星位"              → 第1手 B[pd]
"白棋应以左上角小目"              → 第2手 W[cd]
"黑棋靠入右下角"                  → 第23手 B[nc]
...                               → ...
```

5.5.2 【数量判定】
- 找得出 ≥3 条有效对应 → ✅ 合格，说明对局描写确实基于真实棋谱
- 找不出 ≥3 条 → ❌ 不合格，说明描写太虚（全是"靠""断""夹击"等笼统词），必须重写对局段落

5.5.3 【不合格处理】回到步骤4，用 SGF 坐标重新翻译为自然语言描写，不得使用无坐标对应的笼统表述。

5.5.4 【章节末尾标注】自检通过后，在本章末尾追加参考信息（读者可见）：
```
> 本章对局参考：[SGF文件名] | [黑方名] vs [白方名]([年份]) | 共[N]手
```
此信息仅标注棋谱来源，不含坐标。读者无感，作者可追溯。

**步骤 6 — 回到主线写作**
围棋对局段写完后，返回正常的「继续创作」流程。

## 实战示例

**用户输入：**「写一段男主在围棋比赛中绝境翻盘的描写」

**你应该这么操作：**
1. 确认用户：主角执黑？还是执白？大约多少手的对局？
2. 从 `data/sgf/wu_qingyuan/WuQingYuan_1941_1950_135/` 中找一个白胜且手数在 200+ 的棋谱
3. 用 `read_file` 读取 SGF 头确认对局信息
4. 用 `read_file` 读取全部手数
5. 找到关键转折手（比如 C[] 注释或胜率波动最大的区域）
6. 翻译为小说段落：
   ```
   棋盘上，黑子步步紧逼，白棋形势岌岌可危。
   第 87 手，黑棋在右下角靠入，
   看似要将白棋的大龙就此屠灭。
   所有人都以为胜负已定……
   ```
7. 必要时可以多次 `read_file` 查看不同棋谱比较选择

## 注意事项

1. **永远不要自己编造围棋着法。** 每一手都必须来自真实的 SGF 棋谱。
2. **可以裁剪棋谱长度。** 只需要开局 → 中盘关键转折 → 终局三个片段时，只读棋谱对应位置的手数即可，不必全盘照搬。
3. **可以组合使用。** 写多个对局场景时，每局各选一个真实棋谱，确保局局不同。
4. **名局精解的注释可直接引用。** 但要根据主角和剧情调整语气。
5. **检索不到匹配棋谱时**，告知用户当前棋谱库的覆盖范围，建议更换对局需求。
