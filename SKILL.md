## 📂 文件路径约定

**本 skill 中所有文件引用遵循以下规则：**

1. **skill 内部文件**：`modules/`、`references/`、`templates/`、`memory-system/`（skill 内模板）等路径，均相对于本 skill 的根目录。
   - 示例：`references/书名生成策略.md` → skill 目录下的 references/书名生成策略.md
   - 示例：`记忆系统/bible/story_bible.md`（不带 `[书名]/` 前缀）→ skill 内模板文件，只读

2. **项目文件**：带 `[书名]/` 前缀的路径，相对于当前工作目录。`[书名]` 是项目初始化时用户确认的书名。
   - 示例：`[书名]/记忆系统/bible/story_bible.md` → 项目目录下的记忆系统副本
   - 示例：`[书名]/设定/[书名]_人设卡.md` → 项目目录下的设定文件

3. **记忆系统文件区分**：
   - `memory-system/xxx`（无前缀）= skill 内的只读模板 → 仅用于项目初始化时复制
   - `[书名]/记忆系统/xxx`（有前缀）= 项目目录下的读写副本 → 所有后续操作的目标

4. **项目查找**：跨会话恢复时，在当前工作目录下搜索 `*/记忆系统/bible/story_bible.md` 定位已有项目。

```

   - 分步模式 → 按序执行功能2→3→4→5（世界观→人物→大纲→开篇）

### 场景 B2：写后流水线（内联到 SKILL.md — 每次必读，不得跳过）

> **⚠️ 双代理架构**：本流水线分为主代理任务和子代理任务两部分。
> 主代理负责创作和记忆系统维护（需要写作上下文），子代理负责所有审核和诊断（需要独立上下文，避免自审偏见）。
> 子代理通过 `delegate_task` 并行启动，各自读取磁盘文件完成独立审核。

```
严格按以下步骤顺序执行（「继续」自动触发或用户显式「写后」触发，均不停顿、不询问）：

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  第一阶段：主代理任务（需要写作上下文）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【步骤1】字数确认（脚本校验，不使用LLM计数）
  a. 读取 [书名]/设定/[书名]_立项定位单.md，提取「每章字数」字段
     - 如果读到 → 字数下限 = 该字段值，字数上限 = 字数下限 + 500
     - 如果未读到（旧项目无此字段）→ 字数下限 = 2000，字数上限 = 2500
  b. 执行脚本统计字数：
     python -c "import re,sys; t=open(sys.argv[1]).read(); print(len(re.findall(r'[\u4e00-\u9fff]', t)))" "[书名]/正文/第XXX章_章节名.txt"
  c. 字数 < 字数下限 → ❌ 打回扩写，不进入后续步骤
  d. 字数 > 字数上限 → ❌ 打回精简，不进入后续步骤
  e. 字数达标 → touch [书名]/.done/.done_wordcount

【步骤2】更新4个圣经文档（主代理执行 — 需要理解刚写了什么）
  a. story_bible.md — 更新项目概况（章节数/状态）、当前状态快照（结尾场景/人物在场/最后一句话/主角目标/主要障碍/读者悬念）
  b. character_arcs.md — 更新出场角色的演化时间线（本角色在本章经历了什么、状态变化）、当前状态（位置/情绪/目标/与其他角色的关系）
  c. plot_timeline.md — 在主线时间线追加本章事件记录（含导致/关键人物/备注），更新活跃支线状态
  d. chekhovs_gun.md — 检查并更新伏笔状态（新埋设的伏笔标记为🔘已埋设，已回收的标记为✅已回收）
  e. 全部完成 → touch [书名]/.done/.done_bible

【步骤3】生成单章摘要（主代理执行）
  a. 写入 [书名]/记忆系统/summaries/ch_NNN_summary.md
  b. 内容必须包含：核心事件（3-5句话）、人物变化表、新埋设伏笔、已回收伏笔、衔接锚点、本章结尾状态、下一章方向建议
  c. 已完成 → touch [书名]/.done/.done_summary

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  第二阶段：子代理审核（独立上下文，避免自审偏见）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

> **所有审核任务通过 `delegate_task` 并行启动。**
> 每个子代理的 context 必须包含：
>   - 项目工作目录路径（[书名]/所在目录）
>   - 当前章节文件名
>   - 要求：中文回复，严格按对应模块流程执行，输出结果写入指定文件
> 子代理独立读取磁盘文件，不依赖主代理的写作上下文。
>
> **⚠️ 完全分离规则（适用于所有审核子代理）：**
> - 子代理必须将完整报告写入文件（按模块指定的路径）
> - 子代理只返回一行结论：`PASS` 或 `FAIL: [N]个问题需修复`
> - 主代理不读取审核报告内容，只根据 PASS/FAIL 决定下一步
> - 有 FAIL 时，主代理启动修复子代理处理（见步骤5.1）

【步骤4】并行启动6个审核子代理

  > 每个子代理通过 delegate_task 启动，各自独立读取磁盘文件完成审核。
  > 详细调用规格（goal/context/所需文件/输出格式）写在对应模块文件中。

  使用 delegate_task 的 tasks 数组并行启动以下子代理：

  | 子代理 | 模块文件 | 职责 |
  |--------|----------|------|
  | A — 五道生成门禁 | 内联于本流程（见下方） | 引用/一致性/蓝图/未知实体/描写检查 |
  | B — 强约束检查 | `modules/05_22_强约束系统.md` | 金手指叙事/时间约束/叙事线监控 |
  | C — 读者审稿 | `modules/05_25_读者审稿.md` | 4维度评分（连贯性/吸引力/爽点/Bug） |
  | D — 连贯性检查 | `modules/05_17_功能模块.md` | 人物/剧情/伏笔/世界观连贯性 |
  | E — 节奏诊断 | `modules/05_12_功能模块.md` | 节奏分析+优化建议 |
  | F — 质量检查 | `modules/05_13_功能模块.md` | 情节/人物/爽点/设定质量 |

  子代理A（五道门禁）调用规格：
    goal: "对指定章节执行五道生成门禁检查"
    context: "中文回复。项目目录：{工作目录}，章节文件：{书名}/正文/第XXX章_章节名.txt。
      需要读取：章节正文、人设卡、世界观设定、story_bible.md、character_arcs.md、章节规划表。
      执行五道检查：1.引用 2.一致性 3.蓝图 4.未知实体 5.描写。
      输出：每道 ✅通过 或 ❌不通过+问题+建议。"
    toolsets: ["file"]

  子代理B-F：加载对应模块文件，按其中的「子代理调用规格」执行。
  context 格式统一为：「中文回复。项目目录：{工作目录}，章节文件：{书名}/正文/第XXX章_章节名.txt。按模块流程执行。报告写入指定文件，只返回 PASS 或 FAIL。」

  ⚠️ 子代理必须将完整报告写入文件，只返回一行结论：
  - PASS → 本项审核通过
  - FAIL: [问题数量]个问题需修复 → 有必须修改的问题

【步骤5】收集结论并处理（主代理不读审核报告内容）

  a. 等待6个子代理全部返回

## ⚠️ 自审悖论设计原则

> **核心教训**：同一个模型、同一个对话里"自己写自己审"是无效的。
> 审核结果天然偏向"还行"——因为审核器和写手共享上下文，已经被"洗脑"了。

**已实施的解决方案（v7.1+）**：

1. **结构分离**：所有审核任务（五道门禁/强约束/读者审稿/连贯性/节奏/质量）通过 `delegate_task` 启动独立子代理执行
2. **上下文隔离**：子代理只读磁盘文件，不共享主代理的写作上下文
3. **结果隔离**：子代理只返回 `PASS` 或 `FAIL`，完整报告写入文件。主代理不读报告内容
4. **修复分离**：有 FAIL 时，启动独立修复子代理读报告并修复，而非主代理自己修
5. **字数统计脚本化**：用 Python 脚本统计 CJK 字符数，不让 LLM 数数（又慢又不准）

**不能脚本化的 vs 能脚本化的**：
- ❌ 伏笔判断、逻辑矛盾、描写充分度、节奏评估 → 必须靠模型（所以用独立子代理）
- ✅ 人名/地名/专有名词与设定列表的字符串匹配 → 可以脚本化（但目前未实现，优先级低）

**为什么"换人格"没用**：
- 给审核代理换个"毒舌编辑"语气 → 本质上还是同一个模型同一个上下文
- 人格设定只是表面变化，不解决上下文污染的根本问题
- 结构分离才是正解

**delegate_task 的边界**：
- ✅ 子代理的审核判断是独立的（没有写作偏见）
- ⚠️ 审核结果返回后，主代理的对话里同时有写作上下文+审核结果
- ⚠️ 修复决策还是主代理做的（但至少有了独立的外部输入）
- 完美方案需要审核和修复也分开（已实施：修复由独立子代理执行）

**Pitfall**：子代理 `max_concurrent_children` 限制（默认3），不能一次起超过3个并行子代理。超过时需分批（先3个，完成后再3个）。

**delegate_task 实操要点**：
- 子代理的 `context` 必须包含：项目工作目录路径、章节文件名、需要读取的文件列表
- 子代理用 `toolsets: ["file"]` 限制工具集，减少 token 开销
- 模块文件中应包含完整的「子代理调用规格」（goal/context/所需文件/输出格式），这样主代理只需引用模块名
- 子代理必须将报告写入文件（而非仅返回文本），确保主代理不接触审核细节

## 重要说明

**记忆系统使用方式：**
- **记忆系统**：采用 LLM 原生可读/可写的自然语言文档（故事圣经/人物弧线/剧情时间线/伏笔清单）
- **结构**：`记忆系统/bible/`（4个故事圣经文档）+ `summaries/`（单章摘要）+ `phases/`（10章压缩块）
- **初始化**：项目初始化时自动创建记忆系统
- **自动维护**：每章写完后自动更新圣经 + 生成摘要
- **阶段压缩**：每10章自动将摘要压缩为梗概，确保跨会话记忆不丢失
- **跨会话恢复**：新会话读取圣经 + 压缩块即可秒回状态
- **手动管理**：功能18可查看/编辑/修复记忆系统

**强约束系统：**
- **默认启用**：每章写作后自动检查三个维度（金手指叙事/时间约束/叙事线监控）
- **叙事参谋**：不报违规，只在有风险时给出叙事建议
- **自动初始化**：约束文档不存在时自动创建，无需手动设置
- **手动触发**：`金手指` / `时间约束` / `叙事线` / `强约束`
- **存储位置**：`[书名]/记忆系统/constraints/` 下 3 个追踪文档

**叙事质量管理框架：**
- **写前预检自动触发**：「继续」/「下一章」命令在加载单章写作模块前自动执行写前状态核对
- **阻断与自愈**：缺字段阻断写作 → 自动回溯缺失来源（回读摘要/圣经/弧线）→ 修复记忆系统 → 重新预检通过 → 继续
- **规则集默认**：首次使用自动加载「通用默认」规则集，可通过 `规则集` 切换
- **存储位置**：YAML 链条随章节存档，每 10/50 章压缩至 phases/volumes/

**导出功能：**
- 网文格式：纯txt，适配各平台
- 电子书：epub格式，支持目录导航
- 投稿版：Word格式，标准投稿排版
