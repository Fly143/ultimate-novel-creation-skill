### 17. 连贯性检查 — 全局一致性深度审查

**功能**：对照 v5.0 记忆系统圣经文档，逐项检查 100 章以上长篇的人物/剧情/伏笔/世界观连贯性。生成结构化连贯性报告，标注严重程度并给出修复建议。

---

## 子代理调用规格

> 本模块必须由 `delegate_task` 启动的独立子代理执行。
> 子代理不共享主代理的写作上下文，避免"自己审自己"的偏见。
> 子代理需独立读取：章节正文、5 个圣经文档、最近 10 章摘要。

### 调用方式

```
delegate_task(
  goal: "对指定章节进行全局一致性深度审查，逐项检查人物/剧情/伏笔/世界观连贯性，生成结构化报告",
  context: "中文回复。项目目录：{工作目录}\n\n
    待检查章节文件：{书名}/正文/第XXX章_章节名.txt\n\n
    需要读取的文件：\n
    - 章节正文（待检查对象）\n
    - {书名}/记忆系统/bible/character_arcs.md（人物弧线）\n
    - {书名}/记忆系统/bible/plot_timeline.md（剧情时间线）\n
    - {书名}/记忆系统/bible/chekhovs_gun.md（伏笔清单）\n
    - {书名}/记忆系统/bible/story_bible.md（故事圣经/世界观）\n
    - {书名}/记忆系统/bible/character_database.md（角色数据库）\n
    - {书名}/记忆系统/summaries/（最近 10 章摘要）\n\n
    ⚠️ mode: pipeline — 由05_26流水线调用时自动包含

    读完文件后，严格按以下流程执行连贯性检查：",
  toolsets: ["file"]
)
```

### 需要读取的文件

| 文件 | 用途 |
|------|------|
| 章节正文 | 待检查对象 |
| character_arcs.md | 对照人物性格/能力/关系/对话风格 |
| plot_timeline.md | 对照事件因果/时间线/支线状态 |
| chekhovs_gun.md | 对照伏笔回收/过期/冲突 |
| story_bible.md | 对照世界观规则/势力/物品能力 |
| character_database.md | 对照角色出场/退场状态、能力/持有物变动 |
| 前 10 章摘要 | 检查与近期剧情的连贯衔接 |

### 输出要求

- 将连贯性检查报告写入 `{书名}/设定/第NNN章_连贯性报告.md`（流水线模式）或 `{书名}/设定/连贯性检查报告_第XX章.md`（独立调用模式，含章节编号以区分）
- 如发现必须修复的严重问题（🔴 级别），同时在 `{书名}/记忆系统/bible/` 对应圣经文档中记录冲突条目

## 主检查流程（v5.0 自然语言记忆）

### 步骤 1 — 检查记忆系统是否存在

```
确认 [书名]/记忆系统/ 目录是否存在
  ├─ 不存在 → 提示用户：「记忆系统未初始化，建议先使用功能18初始化记忆系统后再执行连贯性检查」→ 终止
  └─ 存在 → 继续执行
```

### 步骤 2 — 加载记忆上下文

读取以下 5 个圣经文档，建立全局参考基准：

| 文档 | 关键关注点 |
|------|-----------|
| story_bible.md | 世界观规则、势力格局、物品/能力设定、当前状态快照 |
| character_arcs.md | 所有角色的演化线、当前状态、性格特征、能力等级、关系网络 |
| plot_timeline.md | 事件因果链、时间线、主线/支线推进状态 |
| chekhovs_gun.md | 所有伏笔的埋设时间、预期回收时间、当前状态 |
| character_database.md | 所有角色的出场/退场事件、能力/物品持有状态、关系快照 |

同时读取最近 10 章摘要，建立近期剧情的上下文基线。

### 步骤 3 — 人物一致性检查（对照 character_arcs.md）

逐项检查当前章节中出场角色的行为是否与设定一致：

| 检查项 | 方法 | 严重程度标记 |
|--------|------|-------------|
| **性格一致性** | 角色行为是否符合其演化线阶段？有无突变？ | 偏离弧线 → 🔴 / 微调 → 🟡 |
| **能力一致性** | 实力增长是否符合设定速度？是否出现能力倒退未解释？ | 突然变强/变弱 → 🔴 |
| **关系一致性** | 角色关系变化是否有前期铺垫？有无突然和好/决裂？ | 无铺垫突变 → 🟡 / 逻辑矛盾 → 🔴 |
| **对话风格** | 口头禅、语气、用词风格是否与角色设定一致？ | 偶发偏差 → 🟡 / 严重 OOC → 🔴 |
| **OOC 风险标记** | 偏离弧线的行为单独标记，附带偏差描述 | 记录偏差章节号 + 具体行为 |

### 步骤 4 — 剧情连贯性检查（对照 plot_timeline.md）

检查当前章节与已有剧情的时间线和因果关系：

| 检查项 | 方法 | 严重程度标记 |
|--------|------|-------------|
| **事件因果** | 最近事件是否与之前事件逻辑矛盾？新事件是否有合理起因？ | 因果断裂 → 🔴 |
| **时间线** | 时间跨度是否合理？有无时间倒流或无解释跳跃？ | 时间矛盾 → 🔴 |
| **支线状态** | 各支线推进是否合理？有无支线被遗忘超过 50 章？ | 被遗忘支线 → 🟡 |
| **爽点分布** | 对比章节规划表，检查爽点密度是否符合规划 | 密度失衡 → 🟡 |

### 步骤 5 — 伏笔完整性检查（对照 chekhovs_gun.md）

遍历所有伏笔状态，逐项核验：

| 检查项 | 方法 | 严重程度标记 |
|--------|------|-------------|
| **遗漏回收** | 遍历所有未回收伏笔，检查当前章节是否涉及推进 | 遗忘推进 → 🟡 |
| **回收窗口** | 接近或超过回收期限的伏笔 → 标记「即将过期」 | 超期未回收 → 🔴 |
| **矛盾冲突** | 新内容是否与已埋伏笔设定冲突？ | 剧情冲突 → 🔴 |
| **状态确认** | 状态为 🔄（准备回收）的伏笔是否在本章完成回收？ | 未完成 → 🟡 |

### 步骤 6 — 世界观一致性检查（对照 story_bible.md）

检查当前章节与已设定世界观的兼容性：

| 检查项 | 方法 | 严重程度标记 |
|--------|------|-------------|
| **规则冲突** | 新章节是否违反已设定的世界规则？ | 违反规则 → 🔴 |
| **势力变动** | 势力格局变化是否有合理解释和铺垫？ | 无理由变动 → 🟡 |
| **物品/能力一致性** | 关键道具和能力是否出现矛盾描述？ | 描述矛盾 → 🔴 |

### 步骤 7 — 生成连贯性报告

将所有检查结果汇总为结构化报告，写入 `[书名]/设定/第NNN章_连贯性报告.md`（流水线模式）或 `[书名]/设定/连贯性检查报告.md`（独立调用模式）。

## 输出格式

```
=== 连贯性检查报告 ===

【检查范围】：第 X 章（或第 X-Y 章）
【检查时间】：{日期}
【对照文档】：story_bible.md / character_arcs.md / plot_timeline.md / chekhovs_gun.md / character_database.md

【人物一致性】🟢 正常 / 🟡 注意 / 🔴 需修复
✓ 角色A：行为符合第 N 章转折后的性格
✓ 角色B：关系发展有铺垫
⚠️  角色C：第 X 章退场后第 Y 章被提及 → 建议修改
🔴 角色D：能力从第 N 章突然提升未解释 → 需补充修炼/获得契机

【剧情连贯】🟢 正常 / 🟡 注意 / 🔴 需修复
✓ 主线推进逻辑连贯
⚠️  第 X 章与第 Y 章对某物来历描述不一致 → 以 bible 为准修正
✓ 时间线无矛盾
⚠️  支线「XXX」已超过 50 章未推进 → 建议安排推进

【伏笔状态】
已埋伏笔：N 个 | 已回收：M 个 | 待回收：K 个
✓ 伏笔 #X 正常推进中
⚠️  伏笔 #Y 已过回收窗口（计划第 X 章，当前第 Y 章）→ 需处理
🔴 伏笔 #Z 与新内容矛盾 → 需修改新章节或调整伏笔设定

【世界观一致】🟢 正常 / 🟡 注意 / 🔴 需修复
✓ 灵力体系设定无冲突
⚠️  势力「XX」变动缺乏铺垫
🔴 新能力「XX」违反世界规则第 N 条

【修复建议】（按优先级排序）
1. 🔴 [具体问题] → [修复建议]（优先级：高）
2. 🟡 [具体问题] → [修复建议]（优先级：中）
3. 🟡 [具体问题] → [修复建议]（优先级：低）
```

### 严重程度定义

| 等级 | 含义 | 处理要求 |
|------|------|----------|
| 🟢 正常 | 无问题 | 无需处理 |
| 🟡 注意 | 存在潜在风险或轻微偏差 | 建议关注，可在后续修订中处理 |
| 🔴 需修复 | 存在明确矛盾或严重偏差 | 必须在继续创作前修复 |

---

## 检查深度分级

| 模式 | 触发时机 | 检查范围 |
|------|----------|----------|
| **基础检查** | 每次调用（默认） | 遍历 5 个圣经文档，标记明显矛盾 |
| **深度检查** | 每 5 章建议执行（由 05_26 步骤8 自动触发） | 逐章回溯 + 全文交叉验证，生成详细报告 |
| **紧急检查** | 用户说"跑偏了" | 聚焦问题章节 vs 圣经文档，快速定位偏离 |

---

## 工作流 B：统一校验（设定变更后扫描）

**触发时机**：
- 用户修改了设定（人设/世界观/势力等）
- 用户说「检查有没有矛盾」「跑偏了」

**执行步骤**：

### 1. 确定扫描范围
- 默认扫描最近 20 章
- 用户可指定范围（如「检查第 50-100 章」）

### 2. 逐章对照新设定
- 读取变更后的设定文档（更新后的圣经文档）
- 逐章检查正文是否与新设定冲突
- 对照记忆系统摘要快速定位相关章节

### 3. 生成冲突报告

```
=== 统一校验报告 ===

【变更设定】张三能力从「畏火」改为「控火」
【扫描范围】第 45-200 章
【发现冲突】
- 第 45 章：张三说「我从小怕火」→ 与新设定矛盾 → 🔴
- 第 87 章：张三被火系攻击重伤 → 不符合控火能力 → 🔴
- 第 123 章：张三用火把照明 → 可保留（控火者用火合理） → 🟢
- 第 201 章：火焰场景无冲突 → 🟢

【修复建议】
1. 第 45 章：修改张三台词（优先级：高）
2. 第 87 章：调整战斗描写或补充受伤原因（优先级：高）
3. 第 123 章：无需修改
```

### 4. 输出修订清单
- 列出每处冲突的章节号、原文片段、建议修改方向
- 保存至 `[书名]/设定/统一校验报告.md`

## 注意事项

1. **独立执行**：本模块必须通过 delegate_task 由独立子代理执行，确保审查者与创作者身份分离
2. **记忆系统依赖**：依赖 v5.0 记忆系统的 5 个圣经文档，未初始化时应提示用户先执行功能18
3. **报告持久化**：每次检查结果保存为文件，支持历史追溯和对比
4. **与写作流程集成**：在 05_26 写后流水线步骤8（周期检查）中，每5章自动触发本模块的深度检查（复用子代理D的深度模式）
5. **修复后复检**：修复问题后建议重新运行检查，确认修复完整且未引入新问题
