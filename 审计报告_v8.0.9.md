## 🔴 严重问题

### 🔴-1：记忆系统缺失时的阻断逻辑死锁（05_06 vs 05_26）

- **位置**：`modules/05_06_功能模块.md` 第29行 vs `modules/05_26_功能模块.md` 第13-21行
- **描述**：
  - **05_06 步骤0**：记忆系统不存在 → ❌ 阻断，强制要求用户先运行「功能18 工作流A」
  - **05_26 步骤0**：记忆系统任一缺失 → 降级模式，自动调用 05_18-A 初始化，然后重入步骤1
  - **实际调用链**：场景B 中「继续」→ 05_06 步骤0 → 若记忆系统缺失 → 阻断。永远到达不了 05_26 的降级模式（因为 05_06 已经终止了）。
  - 即：05_26 的降级逻辑**永远不可能被触发**——在它之前 05_06 已经硬阻断了。
- **影响**：当项目目录存在但某个 bible 文件意外缺失时，系统不会自动修复，而是报错终止。
- **修复方案**：统一到一处。推荐将 05_06 步骤0 的阻断改为降级（与 05_26 一致）：
  ```
  旧: 如果不存在 → ❌ 阻断
  新: 如果不存在 → 自动执行 05_18 工作流A 初始化，初始化完成后重新检查
  ```

### 🔴-2：`modules/03_` 在模块架构索引中被引用但文件不存在

- **位置**：`SKILL.md` 第60行
- **描述**：SKILL.md 的「模块架构索引」表中列出：
  ```
  | `modules/03_` | （编号预留，当前未使用） | — |
  ```
  但整个技能中没有任何以 `03_` 开头的模块文件。虽然标注了「当前未使用」，但在索引表中列出不存在的东西会引起读者（LLM）困惑——它可能会尝试加载 `modules/03_xxx.md`。
- **影响**：LLM 阅读模块索引时可能去尝试 `read_file("modules/03_xxx.md")` 失败。
- **修复方案**：从索引表中移除该行，或改为注释说明而非表行。

## 🟠 重要问题

### 🟠-1：子代理 context 中占位符不一致（`{书名}` vs `[书名]`）

- **位置**：所有 delegate_task context 模板
- **描述**：SKILL.md 文件路径约定明确使用 `[书名]` 作为占位符，但多个子代理模块的 `context` 模板混用了 `{书名}`：
  - `05_17_功能模块.md` 第18行：`{书名}/正文/第XXX章_章节名.txt`
  - `05_25_功能模块.md` 第22行：`{书名}/正文/第XXX章_章节名.txt`
  - `05_12_功能模块.md` 第24行：`{书名}/正文/第XXX章_章节名.txt`
  - `05_22_功能模块.md` 第33行：`{书名}/正文/第XXX章_章节名.txt`
  - 而 `05_26_功能模块.md` 使用 `{书名}` 在 context 模板中（作为 Python f-string 风格变量）
  - 但 `05_13_功能模块.md` 第20行使用的是 `[书名]`
- **影响**：LLM 在构造 delegate_task 调用时可能混淆占位符格式，导致子代理收到未替换的字面字符串。
- **修复方案**：统一为一种占位符风格。建议全部使用 `[书名]`（与 SKILL.md 约定的项目路径一致），或在每个模块开头明确定义占位符替换规则。

### 🟠-2：`mode: pipeline` 标识在所有子代理 context 中的传递链不完整

- **位置**：`modules/05_26_功能模块.md` 第140行 vs 各子代理模块
- **描述**：
  - 05_26 步骤5 明确要求子代理 C/D/E 的 context 必须包含 `mode: pipeline` 标识
  - 05_25（子代理 C）的「子代理调用规格」中**未列出** `mode: pipeline` 作为 context 的必要组成部分——只在输出要求中提到检测它
  - 05_12（子代理 E）的「调用方式」template context **未列出** `mode: pipeline`，但「输出要求」说要检测
  - 05_17（子代理 D）同样 context 模板未列出 `mode: pipeline`
  - 05_13（子代理 F）的 context 模板也未列出
- **影响**：如果 05_26 调用时正确添加了 `mode: pipeline` 到 context（由主代理手动追加），则没问题。但如果主代理严格按照模块的 `调用方式` template 复制——不含 `mode: pipeline`——子代理将以独立模式运行，输出完整报告而非 PASS/FAIL 一行。
- **修复方案**：在每个子代理模块的「调用方式」context 模板中明确添加 `mode: pipeline` 字段说明，并标注「由 05_26 流水线调用时自动追加」。

### 🟠-3：05_06 步骤0 的「记忆系统不存在→阻断」与 SKILL.md 场景B 流程描述冲突

- **位置**：`modules/05_06_功能模块.md` 第27-29行 vs `SKILL.md` 场景B 第2步
- **描述**：
  - 05_06 步骤0：「记忆系统/ 目录不存在 → ❌ 阻断：请先运行功能18 工作流A」
  - SKILL.md 场景B：「搜索 */记忆系统/bible/story_bible.md 定位项目（找不到 → 转场景A）」
  - 这两条描述的失败模式不同：05_06 说的是"找不到就初始化"，SKILL.md 说的是"找不到就转新开项目"
  - 另外，05_06 检查的是 `[书名]/记忆系统/` 目录存在性，而 SKILL.md 场景B 搜索的是 `*/记忆系统/bible/story_bible.md` 文件——前一步（场景B入口）已经定位到项目，后一步（05_06）再检查目录是否存在，二者的责任边界模糊。
- **影响**：对已有项目来说，如果项目被搜索到但记忆系统目录意外缺失，流程会走 05_06 的阻断而不是 SKILL.md 的"转场景A"，用户体验不一致。
- **修复方案**：将 05_06 步骤0 的阻断改为与 SKILL.md 一致的策略：找不到 → 转场景A（新开项目），而非提示用户手动初始化。

### 🟠-4：05_26 步骤2「主代理逻辑自检」的 PASS 条件缺少明确判定标准

- **位置**：`modules/05_26_功能模块.md` 第66-84行
- **描述**：05_26 步骤2 要求主代理逐条过 4 项自检（设定不自打架/信息越级/前文蓝图一致性/伏笔对齐），任一项不通过 → 主代理直接修正文。但：
  - 没有定义"不通过"的具体判断阈值（如「信息越级」的严重程度分级）
  - 没有说明修正后是否需要重新跑字数校验（修正可能影响字数）
  - 没有说明如果修正无法解决（如蓝图本身就不对）该如何处理
- **影响**：LLM 可能对同一问题产生"通过"和"不通过"两种判断，导致自检流于形式；修正后的字数变化可能突破上下限。
- **修复方案**：
  1. 添加每项自检的 PASS/FAIL 判定标准（如「信息越级」：出现3处以上明确越级 → FAIL）
  2. 明确「修正→重跑字数（步骤1）→若 PASS 继续步骤3」的子循环
  3. 添加「修正3次仍不通过 → 报告用户，暂停流水线」的兜底

### 🟠-5：delegate_task 工具依赖未在任何地方文档化定义

- **位置**：全文多处引用 `delegate_task`
- **描述**：技能依赖 Hermes 的 `delegate_task` 工具来启动独立子代理。但在 Hermes Agent 中，子代理委托使用的是 `terminal` 工具配合 CLI（如 `hermes agent run`），而非一个名为 `delegate_task` 的直接工具调用。SKILL.md 的「自审悖论设计原则」节描述了 `delegate_task` 的概念和边界，但这不是 Hermes 的实际工具名。
- **影响**：LLM 可能尝试调用不存在的 `delegate_task` 工具而失败。
- **修复方案**：在 SKILL.md 的「自审悖论设计原则」中明确实际的委托方式（如使用 `terminal` 后台启动 `hermes agent run` 子代理），或引用 Hermes Agent 文档中的实际子代理调用方法。

## 🟡 建议问题

### 🟡-1：05_24「写前预检」的具体步骤（阻断/自愈/重新预检）分散在多处

- **位置**：`modules/05_24_功能模块.md` 第52-77行 vs `SKILL.md` 场景B 步骤3 vs `modules/00_强制执行协议.md` 步骤C
- **描述**：写前预检的逻辑描述分布在三个文件中：SKILL.md 场景B 说"缺字段阻断并自愈修复"，00_强制执行协议 说"确认预检通过"，05_24 说"阻断→定位→修复→重新预检"。三处描述一致但**05_24 本身没有给出自愈修复的具体步骤细节**（如何回读摘要/圣经/弧线），这些细节隐含在 SKILL.md 的描述中。
- **影响**：LLM 只读 05_24 时不知道具体的自愈修复操作，可能漏步。
- **修复方案**：将自愈修复的 4 步细节（当前时间点缺失→回读摘要→补写；主角状态缺失→回读 arcs→补写 等）从 SKILL.md 移到 05_24 模块中，确保加载 05_24 就能获得完整操作指南。

### 🟡-6：多模块使用 `python -c` 字数统计脚本，但未处理 Python 不可用的降级

- **影响**：半程收尾后可能重复执行某些步骤。
- **修复方案**：在 SKILL.md 或 05_26 中添加 `半程收尾` 的 .done 标记写入规则（写入哪些标记、跳过哪些），确保后续「继续」能正确识别。
