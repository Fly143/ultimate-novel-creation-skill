## 🔴 严重问题（Critical）

### 🔴 C1 — 05_26 G子代理 context 中世界观文件路径错误

- **文件**：`modules/05_26_功能模块.md`
- **行号**：第 105 行
- **问题描述**：
  G（逻辑因果审查）子代理的 `context` 中列出需读取的文件为：
  ```
  需读取：章节正文、设定/世界观设定文档.md、记忆系统/bible/story_bible.md、...
  ```
  路径 `设定/世界观设定文档.md` 是**模板文件名**（`templates/世界观设定文档.md`），而非项目文件名。项目中的实际文件名为 `[书名]_世界观设定.md`（参见 `02_文档存储规范.md` 第 12 行及 `05_02_功能模块.md` 第 30 行输出路径）。
  
  **对比**：同文件中 A 子代理（第 124 行）正确使用了 `{书名}/设定/[书名]_世界观设定.md`，而 G 子代理错误使用了模板名。

- **影响**：G 子代理（每章写后流水线必执行的逻辑因果审查）将以「文件不存在」失败告终，导致逻辑因果审查**全量失效**。该子代理负责空间矛盾/时间矛盾/因果断层/视角纯净 4 项审查，其失效意味着所有章节的读者盲读审核都无法正常执行。

- **修复方案**：
  ```
  - 需读取：章节正文、设定/世界观设定文档.md、记忆系统/bible/story_bible.md、
  + 需读取：章节正文、{书名}/设定/{书名}_世界观设定.md、记忆系统/bible/story_bible.md、
  ```

### 🟠 H2 — 05_26 元数据「完成步骤」字段顺序与实际执行流程不一致

- **文件**：`modules/05_26_功能模块.md`
- **行号**：第 227-228 行（步骤9 元数据块）
- **问题描述**：
  元数据块中 `完成步骤` 字段值为：
  ```
  wordcount/logic_selfcheck/bible/summary/gates/constraints/review/rhythm/quality/coherence/logic_causal
  ```
  实际执行顺序为：
  - 步骤4 第一批：gates(A) → constraints(B) → **logic_causal(G)**
  - 步骤5 第二批：review(C) → **coherence(D)** → rhythm(E)
  - 步骤6 第三批：quality(F)
  
  元数据中 `logic_causal` 被放在最后，`coherence` 在 `quality` 之后，两者均与实际顺序不符。虽然该字段是扁平字符串不存在严格顺序约束，但对于「补缺」流程（通过解析 `完成步骤` 判断断裂点），顺序混乱可能导致误判。

- **影响**：「补缺」命令依赖该字符串判断遗漏步骤时，可能因顺序混乱无法准确定位断裂点。

- **修复方案**：
  调整为实际执行顺序：
  ```
  完成步骤：wordcount/logic_selfcheck/bible/summary/gates/constraints/logic_causal/review/coherence/rhythm/quality
  ```
  同步更新 `02_文档存储规范.md` 第 63 行的同字段。
