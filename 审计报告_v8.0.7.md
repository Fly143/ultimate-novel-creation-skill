## 🔴 严重问题 (3)

### 🔴 S-01: 05_24 自矛盾 —「约束检查结果记入 constraints/」与「不重复写入」冲突

- **位置**: `modules/05_24_功能模块.md` 第 174 行 vs 第 173 行
- **描述**: 第 173 行明确写「三阶读取 22 的数据做二次判断，不重复写入」；但第 174 行写「约束检查结果记入 `[书名]/记忆系统/constraints/`」。两句在同一段落中直接矛盾。
- **影响**: 执行者无法确定手动触发 `复核` 时是否应该写 constraints 文件。如果写了会覆盖 22 子代理 B 的追踪数据；如果不写，「记入」的语义成为空头支票。
- **修复方案**: 删除第 174 行，或改为明确说明：「三阶结果仅记录在 YAML 链条第 5 区块（因果锁）中，不写入 `constraints/` 下的三个追踪文件——这些文件归 22 子代理 B 独占管理。」

### 🔴 S-02: "finish" 工具引用 — 不存在的 Hermes 工具

- **位置**: `modules/02_文档存储规范.md` 第 158 行
- **描述**: 「通过 `finish` 工具交付所有生成的文件」——Hermes Agent 的工具集中不存在 `finish` 工具。
- **影响**: 任何遵循此指令的 agent 在执行时会调用不存在的方法，导致文件无法按预期交付。虽然这不是高频触发点（用户在项目结束后才可能触发），但误导性强。
- **修复方案**: 删除此行，或改为 Hermes 实际可用的交付方式（如报告文件路径、`write_file` 等）。

### 🔴 S-03: 05_06 步骤0 阻断时引用 05_26 前置检查 — 逻辑循环风险

- **位置**: `modules/05_06_功能模块.md` 第 29 行
- **描述**: 「❌ 阻断：记忆系统未初始化，请先运行「功能18 工作流A」或使用「05_26 前置检查」自动修复」。05_26 是**写后**流水线，其步骤0 的降级模式确实包含了调用 05_18-A 初始化记忆系统的能力。但在 05_06 写**前**阶段引用 05_26 前置检查，意味着：先触发写后流水线 → 执行步骤0初始化 → 然后跳过写后流程回到写前。这是一个隐式流程路径，但 05_26 没有「仅执行步骤0然后退出」的机制——它的总流程是顺序不可跳的。
- **影响**: 如果 agent 严格按 `05_26` 的「总流程（顺序不可跳）」执行，会陷入「初始化记忆系统 → 没章节可审（因为还没写）→ 所有子代理 FAIL 或无法执行」的死胡同。如果 agent 足够聪明会跳过，但协议不允许跳。
- **修复方案**: 将 05_26 步骤0 的降级初始化逻辑抽取为独立入口（或在 05_18 工作流 A 中补充），然后 05_06 步骤0 和 05_26 步骤0 都调用此入口。移除 05_06 步骤0 中对「05_26 前置检查」的引用，改为直接调用 05_18-A。
